## Записи

### PD-001. Чистий старт репозиторію

**Дата:** 2026-07-01  
**Статус:** прийнято

**Рішення:** Новий репозиторій з нуля; код з проєкту `agent` не копіюється.

**Чому саме так:** MVP має вузький scope і простішу структуру. Копіювання старого коду тягне зайві залежності, патерни та борг, які не потрібні для скелета.

**Альтернативи:**
- Форк або копія репозиторію `agent`
- Спільний monorepo з `agent`

**Чому відмовилися:** `agent` ширший за потреби MVP; рефакторинг копії дорожчий за свідомий чистий старт з успадкуванням лише ідей (агенти, orchestrator, LLM-шар).

---

### PD-002. Scope MVP: login → prep → кімната → звіт

**Дата:** 2026-07-01  
**Статус:** прийнято

**Рішення:** Перший реліз покриває повний вертикальний сценарій: вхід → анкетування → live-співбесіда → фінальний звіт.

**Чому саме так:** Можна пройти end-to-end сценарій і перевірити цінність продукту без побічних фіч.

**Альтернативи:**
- Лише prep без live-кімнати
- Лише кімната з ручним введенням профілів
- Повний функціонал старого `agent`

**Чому відмовилися:** Prep-only не демонструє головну цінність (спільна співбесіда з агентами). Повний `agent` — занадто довго до першої перевірки. Ручні профілі без prep спрощують demo, але не тестують AI-збір даних.

---

### PD-003. Свідомі виключення з MVP

**Дата:** 2026-07-01  
**Статус:** прийнято

**Рішення:** Поза scope: Issues/теми, бібліотека анкет, історія співбесід, відмовні листи, HR control rail, стрімінг LLM, автотести (окрім build + ручний сценарій).

**Чому саме так:** Кожна з цих фіч — окремий продуктовий шар. Для скелета важливіше стабільний happy path, ніж повнота enterprise-функцій.

**Альтернативи:**
- Issues з самого початку (структурована дискусія)
- Збережені профілі для повторного використання
- Стрімінг відповідей агентів у UI

**Чому відмовилися:** Issues і control rail ускладнюють UX і orchestrator. Saved profiles — окрема модель даних і UI. Стрімінг додає складність Socket/UI без критичної цінності для локального MVP.

> **Примітка (2026-07-03):** oMLX знято з виключень як локальний LLM-провайдер для dev — див. PD-019. Стрімінг LLM лишається поза scope.

### PD-004. Повний prep-цикл з підтвердженням профілю

**Дата:** 2026-07-01  
**Статус:** прийнято

**Рішення:** AI веде діалог → сигнал `readyForConfirmation` → користувач підтверджує → LLM генерує JSON-профіль → prep-сесія закривається.

**Чому саме так:** Людина контролює фіксацію даних перед співбесідою; агенти в кімнаті працюють з перевіреною структурою, а не з сирим чатом.

**Альтернативи:**
- Одразу писати JSON після кожного повідомлення
- Ручне редагування полів профілю в формі
- Без підтвердження — автоматичний перехід у кімнату

**Чому відмовилися:** Постійний JSON — шум і помилки парсингу. Ручні форми — інший UX і дублювання prep-чату. Автоперехід без підтвердження ризикує некоректними профілями в live-чаті.

---

### PD-005. Профілі прив'язані до співбесіди

**Дата:** 2026-07-01  
**Статус:** прийнято

**Рішення:** `Profile` зберігається на рівні `Interview`, без окремої бібліотеки `SavedProfile`.

**Чому саме так:** Мінімальна модель даних; один interview — один набір профілів кандидата й компанії.

**Альтернативи:**
- Глобальна бібліотека профілів користувача
- Профіль компанії на рівні HR-акаунта, перевикористання між співбесідами

**Чому відмовилися:** Бібліотека потребує версіонування, вибору профілю в UI і прав доступу. Для MVP достатньо «одна співбесіда — один контекст».

---

### PD-006. Учасники live-кімнати: 2 людини + 3 агенти

**Дата:** 2026-07-01  
**Статус:** прийнято

**Рішення:** У кімнаті одночасно п'ять учасників: HR, кандидат, Candidate Agent, Company Agent, Arbiter.

**Чому саме так:** Агенти представляють інтереси сторін і модерують дискусію; люди залишаються головними спікерами.

**Альтернативи:**
- Лише Arbiter + люди (без side-agents)
- Один універсальний агент замість трьох
- Агенти лише в prep, у кімнаті — тільки люди

**Чому відмовилися:** Без side-agents губиться асиметрія «інтереси кандидата vs компанії». Один агент не розділяє ролі. Кімната без агентів не перевіряє головну гіпотезу продукту.

---

### PD-007. Порядок агентів у кімнаті

**Дата:** 2026-07-01  
**Статус:** прийнято

**Рішення:** Після повідомлення людини: Candidate Agent → Company Agent → Arbiter. Максимум одне публічне повідомлення від кожного агента за хід.

**Чому саме так:** Спочатку позиція кандидата, потім компанії, потім синтез і модерація Arbiter'ом — логічна арка одного «раунду» дискусії.

**Альтернативи:**
- Arbiter → Company → Candidate
- Паралельні виклики агентів
- Динамічний вибір, хто відповідає

**Чому відмовилися:** Arbiter першим не має контексту від сторін. Паралельність ускладнює порядок у чаті та batch на клієнті. Динамічний вибір — зайва логіка для MVP.

> **Примітка:** у плані розробки (README, день 18) тимчасово згадано інший порядок (Arbiter раніше). Для реалізації орієнтуватися на це рішення та design spec.

### PD-008. Prep — HTTP; кімната — Socket.IO

**Дата:** 2026-07-01  
**Статус:** прийнято

**Рішення:** Анкетування через синхронні REST-запити; live-чат — WebSocket (Socket.IO) з async orchestrator на backend.

**Чому саме так:** Prep — request/response з очікуваною паузою на LLM. Кімната — realtime між двома клієнтами і batch повідомлень від агентів.

**Альтернативи:**
- Усе через WebSocket
- Усе через HTTP long-polling
- SSE для стріму відповідей агентів

**Чому відмовилися:** WebSocket для prep — зайва складність. Long-polling гірший UX у кімнаті. SSE/стрімінг свідомо виключено з MVP (PD-003).

---

### PD-009. Технологічний стек

**Дата:** 2026-07-01  
**Статус:** прийнято

**Рішення:** npm workspaces monorepo; Vue 3 + Vite + Pinia; Express + TypeScript; PostgreSQL 16 + Prisma; Socket.IO; JWT.

**Чому саме так:** Знайомий стек команди, швидкий старт, достатньо для локального MVP і Docker Compose.

**Альтернативи:**
- Next.js fullstack
- Python (FastAPI) backend
- SQLite замість PostgreSQL
- Session cookies замість JWT

**Чому відмовилися:** Next.js — інший розподіл frontend/backend для socket-heavy кімнати. Python — розрив з TypeScript на обох кінцях. SQLite гірше для concurrent room + dockerized postgres уже в плані. Session cookies ускладнюють SPA + окремий frontend container.

### PD-010. LLM: Ollama (локально) + LiteLLM (хмара)

**Дата:** 2026-07-01  
**Статус:** переглянуто → **PD-019** (2026-07-03)

**Рішення:** Плагінований `LlmProvider`; перемикання через `LLM_PROVIDER=ollama|litellm` у `.env`. Ollama на хості, не в Docker.

**Чому саме так:** Локальна розробка без API keys; можливість перемкнутися на хмарну модель для якості/швидкості без зміни коду агентів.

**Альтернативи:**
- Тільки Ollama
- Тільки OpenAI-compatible API
- oMLX як третій провайдер у MVP
- Ollama в Docker-контейнері

**Чому відмовилися:** Один провайдер обмежує сценарії тестування. oMLX — поза scope MVP. Ollama в Docker на macOS часто гірша DX (GPU, розмір образу); `host.docker.internal` простіший для dev.

> **Примітка (2026-07-03):** На Дні 2 реалізовано інший стек — див. PD-019. Ідея плагінованого `LlmProvider` і перемикання через `.env` збережена.

### PD-011. Підхід до скелета: vertical slice + модульність

**Дата:** 2026-07-01  
**Статус:** прийнято

**Рішення:** Hybrid: вертикальні зрізи (auth → prep → room → report) + модульні сервіси (`agents/`, `llm/`, `routes/`).

**Чому саме так:** Кожен день плану дає робочий шматок продукту; модулі не злипаються в один файл до кінця проєкту.

**Альтернативи:**
- Big-bang: спочатку вся backend-архітектура, потім UI
- Тільки горизонтальні шари без end-to-end перевірок
- Один файл «на все» для швидкості

**Чому відмовилися:** Big-bang пізно показує цінність. Горизонтальні шари без зрізів — ризик несумісних частин на дні 15+. Монолітний файл не масштабується до 3 агентів і socket.

### PD-012. Мова продукту — українська

**Дата:** 2026-07-01  
**Статус:** прийнято

**Рішення:** UI, промпти агентів, повідомлення про помилки та звіти — українською.

**Чому саме так:** Цільова аудиторія і сценарій співбесід локальні; єдина мова зменшує змішування в промптах і звітах.

**Альтернативи:**
- Англійська для всього
- i18n з першого дня

**Чому відмовилися:** i18n — зайва робота для MVP. Англійська не відповідає заявленому контексту продукту.

### PD-013. Автентифікація: окремі ролі HR і кандидата

**Дата:** 2026-07-01  
**Статус:** прийнято

**Рішення:** Ролі `HR` і `CANDIDATE`; окремі флоу входу/реєстрації; JWT у заголовку `Authorization`.

**Чому саме так:** Різні кабінети, різні дії (створити співбесіду vs приєднатися за кодом); проста RBAC на backend.

**Альтернативи:**
- Один тип користувача + вибір ролі після логіну
- Magic link без пароля
- Спільний логін з перемикачем «я HR / я кандидат»

**Чому відмовилися:** Вибір ролі на одному акаунті плутає модель даних. Magic link — зайва інфра для локального MVP. Перемикач ролі на одній сторінці гірший UX, ніж окремі entry points.

### PD-014. Приєднання кандидата за 6-символьним кодом

**Дата:** 2026-07-01  
**Статус:** прийнято

**Рішення:** HR створює співбесіду → унікальний код (6 символів) → кандидат вводить код для join.

**Чому саме так:** Простий офлайн-обмін між HR і кандидатом без email-інвайтів у MVP.

**Альтернативи:**
- Email-запрошення з посиланням
- Пошук співбесіди за назвою компанії
- QR-код

**Чому відмовилися:** Email/посилання потребує пошти, шаблонів, терміну дії URL. Публічний пошук — витік даних. QR — зайвий UI без мобільного фокусу.

### PD-015. Одна активна співбесіда на HR і на кандидата

**Дата:** 2026-07-01  
**Статус:** прийнято

**Рішення:** У MVP користувач не веде паралельно кілька активних інтерв'ю.

**Чому саме так:** Спрощує дашборд, статуси і перевірки «чи зайнятий кандидат».

**Альтернативи:**
- Необмежена кількість паралельних співбесід
- Черга співбесід у UI

**Чому відмовилися:** Паралельні інтерв'ю — складні state machine і room routing. Черга — фіча наступних ітерацій.

### PD-016. Завершення співбесіди: ініціатива HR

**Дата:** 2026-07-01  
**Статус:** прийнято

**Рішення:** Arbiter може запропонувати завершення; фінальне рішення — за HR (`POST /interviews/:id/end` → генерація звіту).

**Чому саме так:** HR — сторона найму; кандидат не повинен одноосібно закривати процес із звітом hire/maybe/reject.

**Альтернативи:**
- Автозавершення після пропозиції Arbiter
- Завершення за згодою обох людей
- Таймаут без участі HR

**Чому відмовилися:** Автозавершення ризикує передчасним звітом. Двостороння згода ускладнює UX. Таймаут — непрозора бізнес-логіка для MVP.

### PD-017. Порядок розробки: спочатку HR, потім кандидат

**Дата:** 2026-07-01  
**Статус:** прийнято

**Рішення:** 22-денний план: інфра → HR-кабінет (повний цикл) → кабінет кандидата → спільна кімната → звіт і Docker.

**Чому саме так:** HR створює співбесіду й код; без HR-флоу немає що тестувати з боку кандидата. Патерни prep (чат, профіль, confirm) відпрацьовуються на Company Agent і копіюються на Candidate Agent.

**Альтернативи:**
- Паралельна розробка HR і кандидата
- Спочатку кімната з mock-профілями
- Backend-first без UI до дня 15

**Чому відмовилися:** Паралельність дублює роботу над prep UI. Mock-профілі відкладають перевірку prep-агентів. Backend-only пізно ловить UX-проблеми.

### PD-018. Розгортання: Docker Compose

**Дата:** 2026-07-01  
**Статус:** прийнято

**Рішення:** `docker compose up` піднімає postgres + backend + frontend; LLM (omlx) залишається на хості, не в Docker.

**Чому саме так:** Один стандартний спосіб запуску для перевірки MVP; ізоляція залежностей БД і сервісів.

**Альтернативи:**
- Локальний запуск без Docker
- Kubernetes / хмарний deploy у MVP
- Managed DB (RDS тощо)

**Чому відмовилися:** Без Docker складніше відтворити середовище. K8s — overkill. Managed DB — зайва вартість і налаштування для скелета.

---

### PD-019. LLM: oMLX (локально) + Gemini (хмара)

**Дата:** 2026-07-03  
**Статус:** прийнято

**Рішення:** Плагінований `LlmProvider`; перемикання через `LLM_PROVIDER=omlx|gemini` у `.env`.

- **Локально:** oMLX (`omlx serve`), OpenAI-compatible API `/v1/chat/completions`; модель за замовчуванням `Qwen2.5-7B-Instruct-4bit`; опційно `OMLX_API_KEY` при auth на сервері.
- **Хмара:** Google Gemini через офіційний SDK `@google/generative-ai` (не LiteLLM); модель за замовчуванням `gemini-2.0-flash`.
- oMLX запускається окремим процесом на хості, не в Docker. API keys не комітити.

**Чому саме так:** На машині розробника вже є oMLX з готовою моделлю на Apple Silicon; OpenAI-compatible API спрощує клієнтський код. Gemini напряму — менше залежностей на День 2; інтерфейс `LlmProvider` готовий до додавання LiteLLM/Ollama пізніше без зміни агентів.

**Альтернативи:**
- Залишити PD-010 (Ollama + LiteLLM) без змін
- LiteLLM як проксі до Gemini замість офіційного SDK
- Тільки oMLX без хмарного fallback
- Ollama в Docker-контейнері

**Чому відмовилися:** Ollama потребувала б окремого setup; LiteLLM — зайвий шар для першого вертикального зрізу. Тільки oMLX обмежує demo без локальної GPU. oMLX у Docker на macOS — гірша DX, ніж `omlx serve` на хості.

**Реалізація:** День 2 — `backend/src/llm/`, `POST /api/llm/complete`, `npm run llm:test`. День 3 — endpoint захищено `requireAuth` + `requireHr`.
