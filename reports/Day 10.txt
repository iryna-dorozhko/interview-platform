КОРОТКИЙ ПІДСУМОК

Сьогодні дослідження Cursor ACP з попереднього дня доведено до повної
реалізації. У backend додано новий `LLM_PROVIDER=cursor-acp`, який використовує
локальний Cursor CLI через команду `agent acp`.

Провайдер зберігає чинний контракт `LlmProvider.complete()`: один довгоживучий
ACP-процес повторно використовується протягом життя backend, а кожен completion
отримує окрему ACP-сесію. Це ізолює паралельні розмови Arbiter, Company,
Candidate та FinalReport і не змішує їхні stream chunks.

Особливу увагу приділено безпеці: ask mode, заборонені filesystem/terminal
capabilities, автоматична відмова permissions та інтерактивних запитів,
контрольований runtime directory, MCP preflight і allowlist змінних оточення.

Реалізацію перевірено unit/integration suite, CommonJS build, реальними
послідовними й паралельними ACP-викликами та HTTP-запитом через backend.

У репозиторії за день — 10 комітів, 21 змінений файл,
4945 insertions / 18 deletions.

================================================================================
ЩО БУЛО ДОСЛІДЖЕНО
================================================================================

- Перевірено Cursor CLI `2026.07.09-a3815c0` та ACP v1.
- Підтверджено транспорт: UTF-8 NDJSON / JSON-RPC 2.0 через stdin/stdout.
- Підтверджено startup flow:
  `initialize → authenticate(cursor_login) → session/new`.
- `sessionId` генерує Cursor; клієнт перевіряє його на непорожність і дублікати.
- Підтверджено modern `configOptions` і fallback на legacy `modes`.
- Текст надходить через `agent_message_chunk`, завершення —
  `stopReason: "end_turn"`.
- Поточний Cursor CLI не оголошує `session/close`.
- `temperature` і `maxTokens` не підтримуються ACP та свідомо ігноруються.
- Офіційний `@agentclientprotocol/sdk` є ESM-only, тому для CommonJS backend
  реалізовано вузький внутрішній transport без нової SDK-залежності.

# РЕАЛІЗАЦІЯ BACKEND
################################################################################

================================================================================
1. КОНФІГУРАЦІЯ
================================================================================

- `LLM_PROVIDER` тепер приймає `cursor-acp`.
- Додано `backend/src/llm/cursor-acp.config.ts`.
- Безпечні defaults:
  - `CURSOR_ACP_EXECUTABLE=agent`
  - startup timeout — 15 секунд
  - prompt timeout — 120 секунд
  - shutdown grace — 5 секунд
  - terminate grace — 2 секунди
  - recycle після 100 сесій
  - максимум NDJSON-рядка — 1 MiB
- Числові значення перевіряються як додатні цілі числа.
- `CURSOR_ACP_CWD` дозволяється лише як абсолютний шлях.
- За замовчуванням використовується приватний runtime directory у `os.tmpdir()`.
- Дочірній процес отримує лише allowlist змінних оточення.
- `DATABASE_URL`, `JWT_SECRET` та інші backend-секрети не передаються.
- Перед стартом перевіряються project/user `.cursor/mcp.json`; налаштовані
  MCP-сервери блокують запуск із зрозумілою помилкою.
- Оновлено `backend/.env.example`; токени або credentials у git не додавалися.

================================================================================
2. ACP PROTOCOL CODEC
================================================================================

- Додано `backend/src/llm/cursor-acp.protocol.ts`.
- Реалізовано framing і декодування NDJSON.
- Додано runtime validation JSON-RPC envelopes, request/response IDs,
  initialize/auth/session results, updates і stop reasons.
- Некоректний UTF-8, JSON, schema або завеликий рядок завершують активну роботу
  protocol error, а не залишають transport у невизначеному стані.
- Повідомлення `system`, `user`, `assistant` серіалізуються в один JSON transcript
  зі схемою `interview-platform.chat.v1`.
- Перед transcript додається фіксована інструкція, що `content` є даними,
  а не способом змінити ролі чи структуру prompt.

================================================================================
3. CURSOR ACP CLIENT
================================================================================

- Додано `backend/src/llm/cursor-acp.client.ts`.
- Один provider володіє одним довгоживучим процесом `agent acp`.
- Lazy initialization використовує одну спільну promise: паралельні callers
  не запускають дублікати процесу.
- Кожен `complete()` створює окрему сесію.
- Stream updates маршрутизуються за `sessionId`; глобального response buffer немає.
- Збираються лише `agent_message_chunk` з `content.type === "text"`.
- Thought, tool, plan, usage та інші updates не потрапляють у результат.
- Збір chunks завершується точно на межі prompt response; late chunks ігноруються.
- Перед prompt режим `ask` має бути позитивно підтверджений.
- Усі permission requests відхиляються через фактичний `optionId`:
  `reject_always → reject_once → cancelled`.
- `cursor/ask_question` повертає `skipped`, `cursor/create_plan` — `rejected`.
- Невідомі blocking requests отримують JSON-RPC `-32601`.

================================================================================
4. TIMEOUT, RECOVERY І SHUTDOWN
================================================================================

- Startup і prompt мають окремі таймаути.
- На prompt timeout надсилається `session/cancel`.
- `session/close` викликається лише коли capability справді оголошена.
- Через відсутність close у поточному Cursor CLI процес контрольовано
  перезапускається після ліміту завершених сесій і лише коли активних сесій немає.
- Падіння child process відхиляє всі активні completions, очищає внутрішній стан
  і дозволяє lazy restart під час наступного виклику.
- Shutdown ідемпотентний: cancel → stdin close → normal exit → SIGTERM → SIGKILL.
- Виклик `close()` до першого completion не запускає процес.
- Помилки розділено на executable missing, authentication, protocol, timeout,
  unexpected exit та empty response.
- Generic JSON-RPC `-32000` більше не вважається автоматично auth-помилкою:
  класифікація враховує зміст повідомлення.

================================================================================
5. LLM PROVIDER І FACTORY
================================================================================

- Додано `backend/src/llm/cursor-acp.provider.ts`.
- Сигнатура `complete(messages, options?)` не змінена.
- До `LlmProvider` додано лише optional `close?(): Promise<void>`.
- Factory вміє створювати `cursor-acp`, але не кешує singleton.
- Власником одного спільного provider є `server.ts`.
- Той самий екземпляр передається в усі LLM-dependent routers та orchestrator.
- Інші провайдери — OMLX, Gemini та OpenAI — залишилися stateless.

================================================================================
6. GRACEFUL BACKEND LIFECYCLE
================================================================================

- Додано `backend/src/server-lifecycle.ts`.
- `SIGINT` і `SIGTERM` використовують одну shutdown promise.
- Порядок cleanup:
  1. припинити приймати HTTP/Socket.IO traffic;
  2. закрити room orchestrator і його timers;
  3. закрити LLM provider та активні ACP sessions;
  4. від'єднати Prisma і pg pool.
- Повторний signal не запускає другий shutdown.
- Cleanup продовжується навіть після помилки окремого ресурсу, але виставляє
  failing exit code.

ПРОДУКТОВІ МЕТРИКИ ТА ПРОГНОЗ ВПЛИВУ CURSOR ACP
================================================================================

Сьогодні продуктові метрики не досліджувалися. Виконані перевірки були
технічними: коректність протоколу, стабільність паралельних сесій, успішність
тестів, build, реальні ACP completions і один HTTP-запит. Даних про поведінку
HR і кандидатів, тривалість повної співбесіди, відмови користувачів або якість
рішень у реальних сценаріях сьогодні не збирали.

Тому наведені нижче очікування є гіпотезами, а не виміряними результатами.

1. УСПІШНІСТЬ ЗАВЕРШЕННЯ СПІВБЕСІД

Прогноз: може зрости, якщо Cursor Auto стабільніше за локальну модель генерує
валідні відповіді агентів і JSON фінального звіту. Менша кількість fallback,
timeout і зламаних agent turns має збільшити частку інтерв'ю, що доходять від
READY/LIVE до ENDED із готовим FinalReport.

Ризик: нестабільність Cursor CLI, fair-use обмеження або довгі ACP-відповіді
можуть, навпаки, збільшити частку перерваних сесій.

2. ЧАС ДО ЗАВЕРШЕННЯ ТА LATENCY

Прогноз: повторне використання одного `agent acp` процесу має зменшити startup
overhead порівняно із запуском нового процесу на кожен запит. Водночас ACP є
агентним каналом, тому end-to-end latency одного ходу може бути вищою, ніж у
прямого Gemini API. Це може збільшити тривалість співбесіди й час очікування
між повідомленнями.

3. КОНВЕРСІЯ ТА ВІДМОВИ КОРИСТУВАЧІВ

Прогноз: кращі, змістовніші відповіді агентів можуть підвищити частку кандидатів,
які завершують підготовку та співбесіду. Але помітні паузи між повідомленнями
можуть збільшити abandonment у live room. Напрям впливу без реальних замірів
невідомий.

4. ЯКІСТЬ ФІНАЛЬНОГО ЗВІТУ

Прогноз: Cursor Auto може зменшити кількість невалідного JSON і fallback-звітів,
а також зробити strengths, risks і recommendation змістовнішими. Це потенційно
підвищить довіру HR до звіту та частку звітів, які HR використовує для рішення.

Ризик: coding-agent поведінка й відсутність справжньої system-role семантики в
ACP prompt можуть давати менш передбачувані висновки. Якість потрібно оцінювати
на однакових transcript, а не за одиничним успішним smoke test.

5. ВАРТІСТЬ ОДНІЄЇ ЗАВЕРШЕНОЇ СПІВБЕСІДИ

Прогноз: для локальної розробки та демо Cursor ACP може зменшити прямі витрати
на Gemini/OpenAI, оскільки використовує наявну Cursor subscription. Проте це не
означає нульову продуктову собівартість: слід враховувати subscription,
fair-use limits, час очікування, повторні запити та можливі fallback-провайдери.

6. НАДІЙНІСТЬ

Прогноз: окремі ACP-сесії, timeout, lazy restart і graceful shutdown мають
зменшити технічні помилки та змішування відповідей між інтерв'ю. Очікуваний
позитивний вплив — нижчий LLM error rate і менше незавершених agent turns.
Фактичний рівень надійності стане зрозумілим лише після серії повних E2E-прогонів.

Мінімальний план вимірювання:

- порівняти Cursor ACP і Gemini на однаковому наборі transcript;
- вимірювати p50/p95 latency кожного LLM-виклику та повної співбесіди;
- рахувати completion rate: created → READY → LIVE → ENDED → FinalReport;
- рахувати LLM error, timeout, empty response і FinalReport fallback rate;
- оцінювати частку валідних звітів та узгодженість recommendation із рішенням HR;
- рахувати витрати провайдера на одну успішно завершену співбесіду;
- окремо відстежувати abandonment під час довгих пауз у live room.

РЕФЛЕКСІЯ
================================================================================

Що сьогодні було найважливішим?

  ACP перестав бути лише дослідженням і став повноцінним `LlmProvider`, який
  працює через той самий контракт, що OMLX, Gemini та OpenAI. Це дозволило
  підключити Cursor до всіх агентів без переписування бізнес-логіки.

Яке рішення було найскладнішим?

  Баланс між повторним використанням одного процесу та ізоляцією completions.
  Один процес зменшує startup latency, але вимагає окремих сесій, точного
  routing за `sessionId`, timeout/recovery і контрольованого recycle через
  відсутність `session/close`.

Що зробили особливо правильно?

  1. Не передали child process увесь `process.env`.
  2. Не покладалися лише на `mcpServers: []`, а додали MCP preflight.
  3. Не вигадували підтримку `temperature` і `maxTokens`.
  4. Відхилили не лише file/terminal permissions, а й усі blocking interactive
     requests.
  5. Перевірили не тільки unit-тести, а й реальний Cursor ACP та HTTP flow.

Що варто пам'ятати далі?

  Cursor ACP добре підходить для локальної розробки й демо через Cursor
  subscription, але це агентний CLI-канал, а не production chat API.
  Для production слід окремо оцінювати latency, privacy кандидатських даних,
  fair-use обмеження, спостережуваність і стабільність Cursor CLI.
