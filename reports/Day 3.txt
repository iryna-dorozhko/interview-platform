  Company Agent prep chat — промпт, parseAgentReply, POST /message, seed

Додано implementation plan у docs/superpowers/plans/.


2. BACKEND — COMPANY AGENT

- **Промпт** (`backend/src/agents/prompts/company-agent.uk.ts`) — українською:
  питання про вакансію, вимоги, культуру; агент вітається першим.
- **`parseAgentReply()`** — витягує маркер `READY:true|false` з кінця відповіді
  LLM і повертає `{ message, readyForConfirmation }` без маркера в тексті.
- **`buildCompanyAgentMessages()`** — system prompt + історія з БД, мапінг
  `HUMAN_HR` → `user`, `AGENT_COMPANY` → `assistant`.
- **Маршрут** `POST /api/prep/:interviewId/message`:
    • `message` необов'язковий — порожній POST запускає привітання агента;
    • lazy-створення `PrepSessionHr` при першому зверненні;
    • збереження повідомлень у `PrepMessageHr`;
    • захист `requireAuth` + `requireHr` + перевірка власника співбесіди.
- **Prep router** підключено до сервера (`createPrepRouter`).
- **Seed:** тестова співбесіда (`joinCode=TEST01`) разом із HR-користувачем;
  `seedHrUser` повертає `id` для ланцюжкового seed.
- Unit-тести: `company-agent.test.ts`, `prep.test.ts` (mock Prisma + provider).


3. ДОКУМЕНТАЦІЯ

- DoD Дня 4 позначено виконаним у README.

ЩО БУЛО ЗРОБЛЕНО (ДЕНЬ 5)

Реалізовано **в той самий календарний день**, одразу після закриття Дня 4.

1. ПРОЄКТУВАННЯ (DESIGN SPECS)

Написано й затверджено дизайн-документ у docs/superpowers/specs/:

  Company prep chat UI — сторінка чату, delete/finish, profile extraction

Дизайн свідомо об'єднав День 5 (UI) і частину Дня 6 (генерація `CompanyProfile`
через `POST /finish`), залишивши явне підтвердження профілю (`confirmedAt`) на
День 7.

Додано implementation plan у docs/superpowers/plans/.


2. BACKEND — ДОДАТКОВІ PREP ENDPOINTS

- **`GET /api/prep/:interviewId`** — стан анкети: `messages[]`, `isClosed`,
  `profile | null`; без створення сесії, якщо її ще немає.
- **`POST /api/prep/:interviewId/finish`** — LLM-екстракція структурованого
  профілю з історії чату → `CompanyProfile` (role, requirements, culture,
  expectations); закриває сесію (`isClosed: true`).
- **`DELETE /api/prep/:interviewId`** — повний рестарт: видаляє повідомлення,
  сесію і профіль.
- **`GET /api/interviews/mine`** — список співбесід поточного HR (для навігації
  на найновішу).
- **`buildProfileExtractionMessages()`** + **`parseProfileExtraction()`** —
  окремий промпт для JSON-екстракції профілю з транскрипту діалогу.
- **`ProfileExtractionError`** — валідація структури відповіді LLM.


3. FRONTEND — СТОРІНКА АНКЕТИ

- **`CompanyPrepView.vue`** (`/prep/:interviewId`):
    • завантаження стану через `GET /api/prep/:id`;
    • автоматичне привітання агента при першому відкритті (порожній POST);
    • список повідомлень, поле вводу, індикатор завантаження;
    • кнопка «Завершити чат» — `POST /finish`, показ профілю;
    • кнопка «Видалити чат» — `DELETE`, повний рестарт;
    • режим перегляду історії після закриття (read-only профіль).
- **API-клієнти:** `frontend/src/api/prep.ts`, `frontend/src/api/interviews.ts`.
- **`HomeView.vue`** — кнопка «Анкета компанії» → `GET /interviews/mine` →
  redirect на `/prep/:interviewId` (найновіша співбесіда HR).
- **Router:** маршрут `/prep/:interviewId` з `meta.requiresAuth: true`.

================================================================================
ЧОМУ ДНІ 4–5 НАБЛИЗИЛИ MVP
================================================================================

До сьогодні HR міг лише спілкуватися з «голим» LLM через тестовий чат на
головній. Після Днів 4–5 з'явився **перший повноцінний бізнес-сценарій prep**:

  **HR логіниться → відкриває анкету компанії → веде структурований діалог з
  AI-агентом → історія зберігається в БД → може завершити чат і отримати
  структурований профіль вакансії.**

Це фундамент для наступних кроків HR-флоу:
  • День 7 — підтвердження профілю (`confirmedAt`);
  • День 8 — створення співбесіди HR (замість seed);
  • День 9 — дашборд HR;
  • Candidate Agent (Дні 11–13) — той самий патерн prep + UI.


================================================================================
ПРИЙНЯТІ РІШЕННЯ (ДНІ 4–5)
================================================================================

АРХІТЕКТУРНІ

1. Prep — HTTP (за PD-008), не WebSocket; кожне повідомлення — окремий запит.
2. Lazy-створення `PrepSessionHr` — сесія з'являється при першому POST /message,
   GET не створює запис у БД.
3. Маркер `READY:true|false` в кінці відповіді агента — сигнал готовності до
   завершення без окремого класифікатора.
4. Об'єднання Дня 5 UI і генерації профілю (День 6) в один дизайн — HR одразу
   бачить повний цикл «чат → профіль», без проміжного стану «чат є, профілю немає».
5. Навігація через `GET /interviews/mine` — кнопка на головній веде на
   найновішу співбесіду HR (поки немає дашборду, День 9).
6. Тестова співбесіда в seed — обхід відсутності `POST /interviews` (День 8).

ТЕХНІЧНІ

7. `parseAgentReply` — regex з толерантністю до `[READY:true]`, `(READY:false)`,
   регістру і зайвих пробілів (моделі форматують маркер по-різному).
8. Окремі промпти для діалогу (`company-agent.uk`) і екстракції профілю
   (`company-profile-extraction.uk`) — різні задачі, різні system prompts.
9. `ProfileExtractionError` + 502 при невалідному JSON від LLM — не зберігати
   частковий профіль.
10. Ізоляція помилок БД і LLM у prep route — DB write failure → 500, LLM failure
    → 503/502; не змішувати в один catch.
11. `isClosed` на `PrepSessionHr` — після finish нові повідомлення → 409.

ПРОЦЕСНІ

12. Design spec → implementation plan → код — той самий workflow, що й Дні 1–3.
13. День 4 верифікується curl; День 5 — браузер + curl для нових endpoints.


================================================================================
ПРОБЛЕМИ ТА ЇХ ВИРІШЕННЯ
================================================================================

1. GEMINI ВИМАГАЄ ОСТАННЄ ПОВІДОМЛЕННЯ ВІД USER

   Company Agent вітається першим — при порожній історії останнє повідомлення
   було б від assistant, і Gemini кидав помилку «last message must be from user».

   **Рішення:** у `buildCompanyAgentMessages()` додано placeholder
   `(порожнє повідомлення)` як синтетичний user-turn; у `gemini.provider.ts` —
   `buildGeminiHistory()` з prepend `(start)` якщо history починається з model.

2. МОДЕЛЬ ФОРМАТУЄ МАРКЕР READY У ДУЖКАХ

   Локальна модель (omlx/Qwen) інколи повертала `[READY:false]` замість
   `READY:false`, і маркер потрапляв у текст повідомлення для HR.

   **Рішення:** розширено regex у `parseAgentReply` — підтримка `[]`, `()`,
   відсутності newline перед маркером. Додано unit-тести на всі варіанти.

3. ПОМИЛКИ БД ЗАМАЩОВУВАЛИ LLM-ПОМИЛКИ

   У prep route один catch-блок перехоплював і LLM, і Prisma-помилки —
   складно діагностувати, який шар впав.

   **Рішення:** окремі try/catch для provider init, LLM complete, DB persist.

4. SEED НЕ ПОВЕРТАВ ID HR-КОРИСТУВАЧА

   Ланцюжковий seed тестової співбесіди потребував `hrUserId`, але
   `seedHrUser` не повертав `id`.

   **Рішення:** `seedHrUser` тепер повертає `{ id, email }`.

5. OPENAI PROVIDER TEST ПОЗА SCOPE

   Файл `openai.provider.test.ts` потрапив у test script, але провайдер ще
   не був готовий до CI.

   **Рішення:** тимчасово виключено з npm test script (коміт c7416c1).

6. СУМІСНІСТЬ ПРОВАЙДЕРІВ ДЛЯ «АГЕНТ ВІТАЄТЬСЯ ПЕРШИМ»

   omlx/OpenAI приймають messages[] з assistant на кінці; Gemini — ні.
   Потрібна адаптація на двох рівнях: agent message builder і gemini provider.

   **Статус:** виправлення закомічено та готове до подальшого використання.

РИЗИКИ (ДНІ 4–5)
================================================================================

ЩО МОЖЕ ЗАТРИМУВАТИ НАСТУПНІ ДНІ

1. **Якість локальної моделі (Qwen2.5-7B-4bit)** — агент може ігнорувати
   формат маркера READY або повертати невалідний JSON при екстракції профілю.
2. **Gemini-специфічні обмеження** — history має починатися з user; потрібна
   адаптація для кожного нового агента, що вітається першим.
3. **omlx як зовнішня залежність** — prep-чат падає з 503 без запущеного omlx
   (або без API key для Gemini/OpenAI).
4. **Немає `POST /interviews`** — HR працює лише з seed-співбесідою; до Дня 8
   потрібен ручний seed або curl.
5. **Profile extraction без retry** — якщо LLM поверне невалідний JSON, finish
   повертає 502; HR має повторити або видалити чат.

ТЕХНІЧНІ РІШЕННЯ ПІД ПИТАННЯМ

1. **Частина Дня 6 в Дні 5** — свідоме прискорення; `confirmedAt` (День 7)
   залишається окремим кроком.
2. **Третій провайдер OpenAI** — додано в factory (WIP), сумісний з omlx API;
   корисно для dev без локальної GPU.
3. **Єдиний LLM_PROVIDER на інстанс** — Company Agent і profile extraction
   ділять одну модель.
