НА СТАРТІ ДНЯ (EXECUTIVE)

1. Що отримав продукт сьогодні
   Діалоги HR ↔ кандидат стали живими (socket + typing), а не лише через
   polling. HR може відхилити заявку з листом у діалог і бачити статуси
   ACCEPTED / ADDITIONAL_MEETING / DECLINED, синхронізовані з decision
   letters. Фінальний звіт дозволяє явне recommendation exception. Кандидат
   після confirm бачить історію prep-чату; у live кандидат відповідає на
   всі відкриті питання після втручання HR одним повідомленням.

2. Головний технічний результат
   Socket-шар для dialogs (join, typing relay, broadcast messages/letters)
   плюс typing у live room. Паралельно — поведінка live-агента після HR
   interrupt (збір усіх відкритих питань → один ANSWER) і WIP по ACP:
   idle/request timeouts, `room:agent-stop`, розширений conductor loop.

3. Головний продуктовий результат
   Закрито розрив «є decision letter, але заявка в match лишається в
   підвішеному стані»: листи HR тепер оновлюють application status і
   створюють діалог. Realtime у діалогах відповідає на сумнів Day 15
   (polling vs socket).

4. Фактичний обсяг дня
   - 48 комітів (47 без merge).
   - ~8,5 тис. доданих / ~0,1 тис. видалених рядків у комітах.
   - 1 feature-гілка змерджена в main (`typing-indicator-realtime-dialogs`).
   - ~874 / ~60 рядків незакоміченого WIP (ACP timeouts, agent stop, deletes).

5. Що планується далі
   Довести WIP live-керування (stop + ACP watchdog) до merge, прогнати
   E2E decision → dialog realtime → application status, і за потреби
   почати agent evaluation framework за вже готовим design/plan.


================================================================================
КОРОТКИЙ ПІДСУМОК
================================================================================

Вчорашній Day 15 відкрив пост-інтерв’ю контур (decision + dialogs + unread).
Сьогодні цей контур став операційно повнішим і «живим»: realtime у тредах,
синхронізація статусів заявок, виняток у рекомендації звіту, прибирання
діалогів per-user, плюс покращення live-поведінки агентів.

П’ять основних блоків дня:

1. **Realtime діалоги + typing** — socket join/typing/broadcast для Dialog,
   typing у live interview, merge гілки.
2. **Заявки ↔ HR-листи** — decline letter agent, ACCEPTED/ADDITIONAL_MEETING,
   sync status з interview decisions, UI модалки відхилення.
3. **Фінальний звіт** — recommendation exception (kind + reason) у БД/API/UI.
4. **Live-агенти** — відповідь на всі відкриті питання після HR interrupt;
   vacancy prep: спочатку роль, seniority лише за потреби.
5. **UX діалогів / кандидата** — per-user hide dialog, prep chat history
   після confirm; design agent evaluation framework (ще без імплементації).

На кінець дня в роботі (незакомічено): ACP idle/request timeouts, кнопка
«Зупинити» агентів, `MAX_CONDUCTOR_STEPS`, cascade delete для interview/
vacancy, оновлення «наступних кроків» у candidate prep prompt.

================================================================================
ЩО ЗРОБЛЕНО
================================================================================

1. LIVE ROOM TYPING
   - Socket relay `room:typing`.
   - Shared typing helper на frontend.
   - Індикатор «співрозмовник друкує» у спільному чаті співбесіди.

2. DIALOG SOCKET
   - Join у кімнату діалогу.
   - Relay typing між HR і кандидатом.
   - Broadcast нових повідомлень після POST.
   - Broadcast decision letters у socket room діалогу.

3. FRONTEND THREAD
   - Realtime messages + typing у `DialogThreadView` / `useDialogThread`.
   - Очищення typing indicator при виході з чату.
   - Фікс відсутніх template tags у `DialogThreadView`.

4. ЗЛИТТЯ
   - Гілка `feature/typing-indicator-realtime-dialogs` змерджена в main.


================================================================================
ЧОМУ ЦЕ ВАЖЛИВО
================================================================================

Day 15 залишив відкритим: чи достатньо polling для unread/діалогів.
Сьогодні messages і typing у тредах уже на socket — діалог відчувається як
месенджер, а не як сторінка з періодичним оновленням. Unread badges можуть
лишатися на polling або підтягнутися до socket окремим кроком.

# 6. WIP НА КІНЕЦЬ ДНЯ (НЕЗАКОМІЧЕНО)
################################################################################

На момент звіту в working tree (~874+/60−, 19 файлів):

1. CURSOR ACP НАДІЙНІСТЬ
   - Request timeout за замовчуванням на ACP RPC.
   - Prompt idle watchdog (скидається на кожному text chunk).
   - Best-effort release session / abort на hung close.
   - Hard timeout більше не ретраїться через `withLlmRetry`
     (щоб не спалювати хвилини в live).

2. LIVE КЕРУВАННЯ АГЕНТАМИ
   - Socket `room:agent-stop` (лише HR).
   - UI: «Спробувати ще раз» + «Зупинити» у HR toolbar.
   - `MAX_CONDUCTOR_STEPS` піднято (довша автономна черга після nudge).
   - Не скидати thinking при superseded generation; скидання candidatePosted
     після нового питання компанії в тому ж loop.

3. CASCADE DELETE
   - Видалення interview знімає decisions / зв’язки dialog messages /
     відв’язує vacancyApplication.
   - Видалення vacancy чистить applications, offer decisions, match scores
     у транзакції.

4. CANDIDATE PREP PROMPT
   - «Що далі?» оновлено під актуальний флоу: match/заявка, код/запрошення,
     live, рішення в «Діалогах».

================================================================================
РЕФЛЕКСІЯ
================================================================================

Що сьогодні було найважливішим?

  Не «ще один великий модуль», а дозрівання вчорашнього пост-інтерв’ю
  контуру: діалоги стали realtime, заявки синхронізуються з рішеннями,
  звіт може пояснити виняток у рекомендації.

Який головний продуктовий висновок?

  Цикл «звіт → рішення → діалог → статус заявки» тепер виглядає як єдиний
  HR-операційний шлях. Без sync статусів і живих діалогів decision letter
  лишався б ізольованою фічею сторінки звіту.

Що зараз обмежує продукт?

  Незакомічений шар керування live (stop + ACP timeouts) і відсутність
  повного ручного прогону всього ланцюга decision/dialogs/applications.
  Eval framework лише в docs.

Що варто пам’ятати далі?

  Day 15 відкрив пост-інтерв’ю; Day 16 зробив його живим і прив’язаним до
  заявок. Наступний логічний крок — стабілізувати WIP live-контролю й
  підтвердити весь контур одним E2E, перш ніж роздувати нові великі фічі.
