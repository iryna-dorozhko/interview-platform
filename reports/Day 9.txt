# ДОСЛІДЖЕННЯ: CURSOR ACP ДЛЯ ПРОЄКТУ
################################################################################

================================================================================
КОНТЕКСТ
================================================================================

Мета розмови: **замінити зовнішню LLM** (Gemini / OMLX / OpenAI) у бекенді
платформи співбесід так, щоб виклики йшли **через Cursor ACP** і витрачали
**підписку Cursor** (локально; зокрема безлімітний Auto).

Окрема плутанина, яку треба було прояснити:

1. **Agent Client Protocol (Zed / JetBrains / Cursor CLI)** — як IDE/клієнт
   спілкується з coding-агентом (JSON-RPC, stdin/stdout). Це те, що дає Cursor.
2. **Agent Communication Protocol (Linux Foundation / BeeAI)** — інший протокол:
   агент як HTTP-сервіс. До Cursor CLI `agent acp` **не** відноситься.


================================================================================
ЩО ТАКЕ ACP «ПО СУТІ»
================================================================================

ACP — спочатку **специфікація** (мова домовленості), і вже потім — **реалізації**:

- Спека: які методи існують (`initialize`, `session/new`, `session/prompt`…).
- Транспорт у Cursor CLI: newline-delimited JSON-RPC над **stdio**.
- Код: SDK / клієнт / `agent acp` — конкретні реалізації правил.

Аналогія з LSP: LSP не є «моделлю мови», а протоколом між редактором і language
server. ACP — те саме між клієнтом і AI-агентом.


================================================================================
ЩО ЦЕ ОЗНАЧАЄ ДЛЯ INTERVIEW PLATFORM
================================================================================

Зараз агенти (Arbiter / Company / Candidate / FinalReport) ходять через:

  LlmProvider.complete(messages) → текст відповіді

Cursor ACP — **не** OpenAI-compatible chat endpoint. Це spawn процесу
`agent acp`, сесія, prompt, стрім оновлень, інколи `request_permission`.

Тобто для проєкту план такий (поки без коду в репо):

1. Новий провайдер `cursor-acp.provider.ts`, який реалізує той самий
   `LlmProvider.complete`.
2. Сесія в режимі **`ask`** (без правок файлів) + обмеження
   `clientCapabilities` (без fs/terminal).
3. Модель **Auto** (`default`) — щоб скористатися безлімітним Auto на плані.
4. Автентифікація: `agent login` / `CURSOR_API_KEY` — білінг іде з **тієї ж
   підписки Cursor**, окремого «ACP-тарифу» немає.
5. Permissions: клієнт може відхиляти tool-запити, якщо агент їх все одно шле.

**Висновок на кінець дня:** для локальних прогонів і демо — **реалістичний**
варіант; для продакшен-бекенду як «просто LLM API» — крихкий (latency, agent
поведінка, fair use). У код сьогодні **не мерджили** ACP-провайдер — лише
дизайн підходу.

РЕФЛЕКСІЯ
================================================================================

Що сьогодні найбільше здивувало?

  Швидкість «день плану + три великі фічі поруч»: після Day 21 одразу пішли
  візуал, список звітів і invitation dual-channel — і все на main.

  Друге — **ACP ≠ дешевий OpenAI**. Хочеться «підписка Cursor замість Gemini»,
  а отримуєш coding-агента з JSON-RPC і правами на інструменти. Для
  interview-оркестратора потрібен саме текстовий `complete()`, тобто треба
  свідомо «обрізати» агента до ask-режиму.

Яке рішення сьогодні було найважчим?

  Архітектура запрошень: окрема модель `Invitation` vs поля на `Interview`.
  Обрали окрему модель (чисті статуси PENDING/ACCEPTED/DECLINED/CANCELLED) —
  складніше в міграції, простіше в UI й lifecycle.

Що, якби починали заново, зробили б інакше?

  1. Prisma generate / migrate перевіряти одразу після merge invitation —
     running server тримав старий client і падав на `scheduledAt` («Unknown
     argument»), хоча міграція в БД уже була.
  2. ACP proof-of-concept (один `complete` з ask+Auto) робити **маленьким
     spike-комітом** у той же день, поки контекст свіжий — не лише conversation.
  3. FinalReport fallback помічати в UI окремим банером, щоб HR не читав E2E-
     нотатки як «справжній» вердикт моделі.
