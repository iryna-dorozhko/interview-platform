КОРОТКИЙ ПІДСУМОК
================================================================================

Сьогодні зроблено чотири продуктові кроки, які зміцнюють підготовку й живий
чат:

1. **Contact bootstrap** у Candidate Agent — збір імені, email і телефону на
   старті анкети; контакти в `CandidateProfile` і в UI профілю.
2. **Глобальний профіль компанії** — окремий чат HR (культура, напрям,
   політики, формат, онбординг), жорсткий gate перед анкетою вакансії,
   snapshot універсальних полів у `CompanyProfile`.
3. **Arbiter Conductor** — структуровані команди (`START` / `ANSWER` /
   `NEXT_QUESTION` / `WAIT` …), conductor-loop в orchestrator, стрічка
   «Рішення Arbiter» для HR.
4. **Live Candidate Agent** — відповіді в третій особі; обґрунтовані
   припущення з анкети + `needsHuman`; Arbiter чекає після підтвердження.

Паралельно в робочому дереві (ще не закомічено) — редагування повного профілю
перед confirm, покращення company/candidate prep UI, тести.

У репозиторії за день — **28 комітів** (~75 файлів у комітах,
~6030 / 581 insertions/deletions) + WIP ~25 файлів.

================================================================================
ПРОДУКТОВІ МЕТРИКИ ТА ПРОГНОЗ ВПЛИВУ
================================================================================

Сьогодні продуктові метрики не вимірювалися на реальних користувачах — лише
дизайн, імплементація, unit-тести та UI-скріншоти. Нижче — гіпотези впливу
сьогоднішніх змін і мінімальний план замірів.

1. ЧАС PREP ДЛЯ HR (COMPANY + VACANCY)

База: раніше HR повторював культуру/формат на кожній вакансії.
Прогноз: час на другу+ вакансію падає — універсальні поля вже в глобальному
профілі; у vacancy prep лишаються role/requirements/expectations.
Метрика: median хвилин від «створити вакансію» до `confirmedAt` профілю;
окремо — час першого company-profile confirm vs наступних vacancy prep.

2. COMPLETION RATE СПІВБЕСІДИ (READY → LIVE → ENDED)

База: live інколи «мовчав», бо агенти обирали `post:false`.
Прогноз: conductor підвищує частку інтерв'ю з ≥N змістовних ходів і з
FinalReport без порожнього transcript.
Ризик: надто довгий loop або часті WAIT можуть збільшити abandonment, якщо
кандидат довго не відповідає на підтвердження припущення.
Метрика: % ENDED з FinalReport; середня кількість публічних agent messages
за сесію; % сесій із ≥1 `room:arbiter-process` подією.

3. ЯКІСТЬ ВІДПОВІДЕЙ AI-КАНДИДАТА (ДЛЯ HR)

База: шаблонний deferral або відповідь від «я» без цінності для HR.
Прогноз: третя особа + припущення з анкети дають HR контекст навіть до
відповіді людини; transcript стає придатнішим для FinalReport.
Ризик: «припущення + перепит» може здаватися HR подвійним повідомленням
(див. розділ «Сумніви»).
Метрика (ручна оцінка на 10 transcript): частка ходів із корисною тезою для
HR до відповіді людини; частка першої особи; частка дослівного шаблону.

4. КОНТАКТНИЙ BOOTSTRAP → ДОСТАВНІСТЬ ЗАПРОШЕНЬ

База: профіль без телефону/імені ускладнює follow-up поза платформою.
Прогноз: вища частка профілів із `fullName` + `email`; `phone` — опційно.
Метрика: % confirmed профілів із non-empty fullName/email/phone; частка
email = auth email (fallback) vs введений кандидатом.

5. ДОВІРА Й ABANDONMENT У LIVE ПРИ DEFERRAL

Прогноз: WAIT після assumption зменшує «агент сам собі відповідає», але
може збільшити час простою, поки людина не підтвердить.
Метрика: median секунд від `needsHuman` до наступного `HUMAN_CANDIDATE`;
% сесій, де після WAIT людина так і не відповіла (abandon mid-deferral).

Мінімальний план вимірювання (завтра / найближчі прогони):

- 3–5 ручних E2E з фіксацією часу prep і кількості agent turns;
- розмітити 10 live-ходів: clear answer / assumption+confirm / bare deferral;
- порівняти 1 сесію «до conductor» (якщо є запис) vs «після»;
- у FinalReport перевірити, чи потрапляють припущення як strengths/risks, а
  не як шум.

# СУМНІВИ: ПРИПУЩЕННЯ + ПЕРЕПИТ КАНДИДАТА
################################################################################

================================================================================
КОНТЕКСТ
================================================================================

Поточний контракт (режим 2): AI-кандидат бачить у профілі прогалину / зону
росту → публікує коротке припущення для HR третьою особою → просить живу
людину підтвердити або доповнити (`needsHuman:true`) → Arbiter робить WAIT.

Приклад з кімнати: питання про REST API → «З анкети виходить, що робота з API
— зона росту… Ірино, підтвердь або доповни.»

================================================================================
ЧОМУ ТАК ЗРОБИЛИ
================================================================================

- HR одразу отримує сигнал з анкети, а не порожнє «я не знаю».
- Кандидат лишається джерелом правди: AI не вигадує і не закриває тему сам.
- Оркестратор уже вміє стопатися на `needsHuman` — мінімальна зміна коду.

================================================================================
СУМНІВ
================================================================================

Чи не виглядає це як подвійна робота: AI вже «відповів», але все одно
зупиняє розмову й змушує кандидата повторити/підтвердити? Для HR це може
бути цінно; для кандидата — дивно («навіщо мене питали, якщо відповідь уже
є?»). Для темпу співбесіди — ризик зайвої паузи на кожній зоні росту.

Альтернативи для рішення керівника:

  A. Як зараз: assumption + confirm (прозорість для HR, стоп до людини).
  B. Assumption без стопу: пост для HR, але `needsHuman:false` і Arbiter
     одразу `NEXT_QUESTION` / `CLARIFY` — швидше, але ризик, що неточне
     припущення «залипне» в transcript і FinalReport.
  C. Лише deferral без тези: коротко «Ірино, відповідай сама» — менше шуму,
     але HR знову бачить порожнечу до відповіді людини.
  D. Два канали: публічно — лише прохання відповісти; у панелі Arbiter /
     прихованому summary — припущення з профілю (HR бачить, кандидат — ні).
  E. Soft confirm: AI публікує припущення й одразу ставить наступне питання;
     кандидат може виправити пізніше (або кнопкою «уточнити») — найменше
     тертя, найвищий ризик «тихої» помилки в звіті.

Попередня думка команди (не затверджено): D або гібрид — A лише при сильній
впевненості (явна зона росту в анкеті), C при слабкому натяку.

Рішення не фіксуємо самостійно. Чернетку альтернатив (A–E) і попередню
рекомендацію команди зафіксовано в звіті; **фінальний вибір — за порадою
керівника**.

Статус: очікуємо фідбек щодо варіанту за замовчуванням і чи
різнити поведінку залежно від сили сигналу в профілі.

РЕФЛЕКСІЯ
================================================================================

Що сьогодні було найважливішим?

  Розділення «універсальне / вакансійне» для HR і «диригент замість мовчання»
  для live-чату. Обидва знімають повторювану біль: HR не переказує культуру
  на кожну вакансію, а агенти не «висять», бо ніхто не бере хід.

Яке рішення було найскладнішим?

  Arbiter Conductor: баланс між автономним loop і контролем людини
  (`WAIT` / `needsHuman`). Занадто агресивний loop — шум; занадто пасивний —
  знову тиша в кімнаті. Друге — саме deferral з припущенням: цінність для HR
  vs відчуття «відповідь уже є» для кандидата.

Що варто пам'ятати далі?

  Snapshot фіксує стан на момент finish вакансії — зміна глобального профілю
  не переписує старі анкети. Це правильно для історії, але HR має розуміти,
  що «оновити культуру скрізь» — окрема операція (її сьогодні немає).
