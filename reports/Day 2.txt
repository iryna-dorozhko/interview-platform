До LLM-інтеграції додано implementation plan у docs/superpowers/plans/.


2. BACKEND — LLM-ШАР

- Інтерфейс `LlmProvider` з методом `complete(messages[])` і типами `ChatMessage`.
- Ієрархія помилок: `LlmUnavailableError` (503), `LlmEmptyResponseError` (502).
- **OmlxProvider** — OpenAI-compatible клієнт до `POST /v1/chat/completions`:
    • модель за замовчуванням: `Qwen2.5-7B-Instruct-4bit`;
    • timeout 120 с (перший запуск моделі може бути повільним);
    • підтримка `OMLX_API_KEY` для серверів omlx з увімкненою auth.
- **GeminiProvider** — обгортка над `@google/generative-ai`:
    • модель за замовчуванням: `gemini-2.0-flash`;
    • `system` → `systemInstruction`, `user`/`assistant` → chat history.
- **Factory** `createLlmProvider()` — перемикання через `LLM_PROVIDER=omlx|gemini`.
- **Маршрут** `POST /api/llm/complete` — приймає `message` або `messages[]`,
  повертає `{ text, provider }`.
- **CLI** `npm run llm:test --workspace backend` — перевірка провайдера без HTTP.
- Unit-тести: omlx.provider, factory, llm route (mock fetch / fake provider).


3. FRONTEND — ЧАТ З AI

- API-клієнт `frontend/src/api/llm.ts` — `sendChat(messages[])`.

    • кнопка «Новий чат»;
    • індикатор провайдера під відповіддю (`omlx` / `gemini`);
    • обробка 503 («Запусти omlx serve») і інших помилок у банері.

НЮАНСИ СТОСОВНО GEMINI (ДЕНЬ 2)
================================================================================

1. ВІДХИЛЕННЯ ВІД ПОЧАТКОВОГО ПЛАНУ (PD-010 → PD-019)

   MVP-дизайн від 1 липня (PD-010) передбачав Ollama + LiteLLM. На День 2
   прийнято практичніше рішення: oMLX + Gemini напряму через
   `@google/generative-ai`. Зафіксовано в Product-Decisions.md як **PD-019**
   (2026-07-03). Інтерфейс `LlmProvider` збережено — LiteLLM/Ollama можна
   додати пізніше як третій провайдер без зміни агентів.

2. ПЕРЕМИКАННЯ ПРОВАЙДЕРА — ЛИШЕ ЧЕРЕЗ ENV

   У UI немає перемикача omlx ↔ gemini. Для хмарної моделі потрібно:
     LLM_PROVIDER=gemini
     GEMINI_API_KEY=<ключ>
     GEMINI_MODEL=gemini-2.0-flash   (опційно, є default)
   Без `GEMINI_API_KEY` factory кидає помилку ще до HTTP-запиту → 503.

3. РІЗНА СЕМАНТИКА API ПОРІВНЯНО З OMLX

   omlx приймає масив `messages` у форматі OpenAI (system/user/assistant) одним
   запитом. Gemini вимагає мапінгу:
     • system-повідомлення → `systemInstruction` у `getGenerativeModel()`;
     • assistant → role `model` у history;
     • останнє повідомлення обов'язково від `user` (інакше throw).
   Це важливо для майбутніх агентів із system prompt: обидва провайдери
   підтримують system, але реалізація різна.

4. ТЕСТУВАННЯ GEMINI

   Unit-тести покривають лише omlx (mock fetch) і factory (вибір провайдера).
   Gemini перевіряється вручну: `LLM_PROVIDER=gemini GEMINI_API_KEY=... npm run
   llm:test --workspace backend`. У CI живий Gemini не викликається.

5. ОПЕРАЦІЙНІ ОБМЕЖЕННЯ

   • Потрібен мережевий доступ і валідний API key Google AI Studio.
   • Квоти, rate limits і вартість запитів — на стороні Google; у коді немає
     retry/backoff.
   • Немає стрімінгу — відповідь чекаємо цілком (як і в omlx).
   • Ключі не комітяться; у `.env.example` лише закоментовані плейсхолдери.

ЧОМУ ДЕНЬ 2 НАБЛИЗИВ MVP
================================================================================

До сьогодні система вміла лише перевіряти «живість» інфраструктури (health).
Після Дня 2 з'явився перший повноцінний AI-сценарій:

  **Користувач (браузер або curl) → backend → LLM → текстова відповідь з
  контекстом попередніх реплік.**

Це фундамент для всього, що далі:
  • Company Agent (День 4) — той самий `LlmProvider`, але з system prompt і
    збереженням у БД замість локального стану Vue;
  • prep-чат HR і кандидата — той самий контракт `messages[]`;
  • live-кімната з трьома агентами — делегування в провайдер без переписування
    HTTP-шару.

Плагінована архітектура дозволяє розробляти локально на omlx (без API keys,
приватність даних) і перемикатися на Gemini для перевірки якості відповідей
або демо без локальної GPU-моделі — одна зміна в `.env`.

ПРИЙНЯТІ РІШЕННЯ (ДЕНЬ 2)

АРХІТЕКТУРНІ

1. oMLX замість Ollama — готова модель на Apple Silicon, OpenAI-compatible API.
2. Gemini напряму (SDK), не LiteLLM — менше залежностей на День 2; інтерфейс
   готовий до розширення.
3. Єдиний endpoint `/api/llm/complete` — мінімальний контракт для UI, curl і CLI.
4. Повна історія `messages[]` у фронтенд-чаті — backend уже підтримує; single-
   message втрачав би контекст діалогу.
5. Локальний стан чату в компоненті — без Pinia/БД на День 2 (persist — День 4+).

ТЕХНІЧНІ

6. Timeout 120 с для omlx — cold start локальної моделі.
7. HTTP 503 при недоступності LLM, 502 при порожній відповіді — зрозумілі коди
   для UI і моніторингу.
8. `OMLX_API_KEY` — Bearer auth, якщо в omlx увімкнено API key authentication.
9. Factory fail-fast без `GEMINI_API_KEY` — не відкладати помилку конфігурації
   до runtime-запиту в продакшені.

ПРОЦЕСНІ

10. Design spec → implementation plan → код — той самий workflow, що й День 1.
11. Endpoint + CLI + UI — три способи верифікації DoD (API, термінал, браузер).

Реалізовано **в той самий календарний день**, одразу після закриття Дня 2.

1. ПРОЄКТУВАННЯ (DESIGN SPECS)

Написано й затверджено дизайн-документ у docs/superpowers/specs/:

  HR auth (День 3) — JWT, login/me, middleware, router guard, Pinia store

Додано implementation plan у docs/superpowers/plans/.


2. BACKEND — АВТЕНТИФІКАЦІЯ

- **`hashPassword`** (`backend/src/auth/password.ts`) — SHA-256, той самий алгоритм,
  що в seed `hr-user.js`; перевірка пароля seed-користувача без змін у БД.
- **JWT** (`backend/src/auth/jwt.ts`) — `signToken` / `verifyToken` через
  `jsonwebtoken`; payload: `sub`, `email`, `role`; термін за замовчуванням 24 год.
- **Middleware** (`backend/src/auth/middleware.ts`):
    • `requireAuth` — читає `Authorization: Bearer`, кладе `req.user`;
    • `requireHr` — дозволяє лише `role === HR`.
- **Маршрути** (`backend/src/routes/auth.ts`):
    • `POST /api/auth/login` (публічний) — email + password → `{ token, user }`;
    • `GET /api/auth/me` (`requireAuth`) — поточний користувач.
- **Захист LLM:** `POST /api/llm/complete` — `requireAuth` + `requireHr`.
- **`GET /api/health`** залишився публічним (bootstrap Day 1).
- Env: `JWT_SECRET` (обов'язковий, мін. 8 символів), `JWT_EXPIRES_IN` (default `24h`).
- Unit-тести: password, jwt, middleware, auth routes.


3. FRONTEND — ЛОГІН І ЗАХИСТ МАРШРУТІВ

- Залежності: `vue-router`, `pinia`.
- **`api/client.ts`** — `fetchWithAuth` з заголовком `Authorization: Bearer`.
- **`api/auth.ts`** — `login()`, `fetchMe()`; `api/llm.ts` переведено на auth-клієнт.
- **Pinia store `auth`** (`stores/auth.ts`):
    • `token` синхронізовано з `localStorage` (ключ `auth_token`);
    • `login`, `logout`, `restoreSession` (при старті → `GET /api/auth/me`).
- **Router** (`router/index.ts`):
    • `/login` → `LoginView` (публічний; якщо вже залогінений → `/`);
    • `/` → `HomeView` (`meta.requiresAuth: true`);
    • guard: без токена → `/login?redirect=...`.
- **`LoginView.vue`** — email, password, обробка 401/403/мережі.
- **`HomeView.vue`** — перенесено вміст колишньої `App.vue` (health + ChatPanel);
  шапка з email HR і кнопкою «Вийти».
- **`App.vue`** — лише `<RouterView />`; `main.ts` підключає Pinia і router.


4. ДОКУМЕНТАЦІЯ

- README: секція «День 3 — Вхід для HR», Auth Quick Start, curl з Bearer token.
- `backend/.env.example`: `JWT_SECRET`, `JWT_EXPIRES_IN`.
- DoD Дня 3 позначено виконаним у README.

ЧОМУ ДЕНЬ 3 НАБЛИЗИВ MVP
================================================================================

До Дня 3 будь-хто з доступом до localhost міг викликати LLM і бачити чат. Після
Дня 3 з'явився перший **захищений HR-сценарій**:

  **HR логіниться → отримує JWT → бачить кабінет → спілкується з AI через
  захищений API; неавторизований користувач не потрапляє ні в UI, ні в backend.**

Це відкриває шлях до HR-кабінету (Дні 4–9):
  • Company Agent і анкета — лише для автентифікованого HR;
  • майбутні `POST /prep/:interviewId/message` — той самий патерн `requireAuth` +
    `requireHr`;
  • кандидатський флоу (пізніше) — окремий login з `role === CANDIDATE` за тим
    самим каркасом JWT + middleware.

Сесія в `localStorage` + `restoreSession` дає звичний UX «увійшов і залишився в
системі» після перезавантаження — без цього prep-чат у браузері був би незручним.


================================================================================
ПРИЙНЯТІ РІШЕННЯ (ДЕНЬ 3)
================================================================================

АРХІТЕКТУРНІ

1. JWT у `Authorization: Bearer` — за PD-013 і MVP-дизайном; stateless для dev.
2. vue-router + Pinia — одразу цільовий стек; composable без store відкладено.
3. Захист і frontend (guard), і backend (middleware) — подвійний контур безпеки.
4. Після логіну → `/` (health + чат) — мінімальний «кабінет» до окремих сторінок

5. Логін лише для HR у Дні 3 — CANDIDATE на login отримує 403.

ТЕХНІЧНІ

6. SHA-256 для паролів (як у seed) — bcrypt відкладено; сумісність з `hr@test.com`.
7. `JWT_SECRET` обов'язковий при старті backend — fail-fast, не працювати без ключа.
8. 401 від `/api/auth/me` при restore → logout і очистка `localStorage`.
9. 401 від будь-якого API під час сесії → logout + редірект на `/login`.

ПРОЦЕСНІ

10. Design spec → implementation plan → код — той самий workflow.
11. Ручний чекліст DoD: логін, чат після логіну, reload, logout, curl без токена.

ЩО МОЖЕ ЗАТРИМУВАТИ НАСТУПНІ ДНІ

6. **Відсутність LLM у health** — health OK, чат може бути недоступний.

2. **Стрімінг** — blocking request/response; для live-кімнати (Дні 14+) потрібен
   stream.
3. **Єдиний LLM_PROVIDER на інстанс** — всі агенти ділять одну модель.
4. **Немає rate limit на login** — brute-force на `/api/auth/login` у dev.
