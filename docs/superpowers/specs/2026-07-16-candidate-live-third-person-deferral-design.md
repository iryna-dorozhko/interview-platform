# Candidate Live Agent: третя особа + припущення з анкети (deferral)

**Дата:** 2026-07-16  
**Статус:** затверджено в чаті (варіант A + третя особа)

## Проблема

У live-чаті AI кандидата:

1. Копіював шаблонну фразу з промпту («Я не знаю відповіді з профілю. Ірино, дай відповідь сама.») замість природного звернення.
2. На питання про Pinia не робив висновку з анкети (прогалина / зона вивчення), а одразу віддавав deferral без змісту для HR.
3. Відповідав від першої особи («я вивчаю…»), хоча має говорити **про** кандидата.

Існуюча механіка `needsHuman:true` + стоп оркестратора + Arbiter `WAIT` залишається коректною і не змінюється на рівні коду оркестратора.

## Цілі

- AI кандидата завжди говорить **про кандидата** (третя особа), не «від імені» в першій особі.
- Якщо з профілю можна зробити обґрунтоване припущення (зокрема про відсутність компетенції) — опублікувати його для HR і попросити живу людину підтвердити/доповнити (`needsHuman:true`).
- Якщо в профілі немає релевантних даних — природно попросити кандидата відповісти (без дослівної шаблонної фрази, без вигаданих фактів).
- Arbiter після такого повідомлення чекає `HUMAN_CANDIDATE` (`WAIT`), не повторює `ANSWER`.

## Нецілі

- Нові поля JSON відповіді Candidate (окремий `needsConfirm` тощо).
- Зміна парсера `parsePostReply` / логіки оркестратора (вже стопається на `needsHuman`).
- Зміна голосу prep-агента кандидата (анкета до live).

## Підхід

**Лише промпти + системні nudge.** Без нових intent-типів і без пост-обробки «я→він/вона».

## Поведінка Candidate Live Agent

### Голос

- Завжди третя особа: «Кандидат має досвід з Vue…», «У кандидата в зоні росту — Pinia…».
- Заборонено першу особу як відповідь за кандидата («Я працював…», «Я вивчаю…»).
- Звернення до живої людини — по імені (якщо відоме з історії/профілю), вибірково й природно.

### Три режими `ANSWER`

| Ситуація | Повідомлення | `needsHuman` |
|---|---|---|
| У профілі є чітка відповідь | Факти з профілю про кандидата для HR | `false` |
| З профілю видно прогалину / «ще вивчає» / відсутність досвіду з теми | Коротка відповідь-припущення для HR + прохання підтвердити або доповнити | `true` |
| У профілі нічого релевантного | Без вигаданих фактів; коротко й природно попросити кандидата відповісти | `true` |

**Приклад (Pinia, прогалина в анкеті):**  
«З анкети виходить, що кандидат ще не застосовував Pinia на практиці й зараз її вивчає. Ірино, підтвердь або доповни.»

**Приклад (немає даних):**  
«Ірино, у профілі немає деталей з цього питання — будь ласка, відповідай сама.»  
(Формулювання вільне; заборонено копіювати старий шаблон дослівно.)

### Жорсткі правила

- `post:true` обов’язково в обох deferral-режимах (не мовчати).
- Заборонено вигадувати досвід, навички, проєкти чи факти поза профілем.
- Припущення дозволені лише як **висновок з профілю** (включно з відсутністю згадки / зоною росту), з явним проханням підтвердити.

## Arbiter

Оновити правило черги:

- Якщо Candidate щойно попросив живу людину відповісти **або** висунув припущення з проханням підтвердити/доповнити — `WAIT` (не повторювати `ANSWER`).
- Решта дій без змін. Оркестратор уже break після `needsHuman:true`.

## Тест-план (ручний)

1. Live: питання про технологію, якої немає в профілі / у зоні росту → AI дає припущення про кандидата + просить підтвердити; арбітер мовчить, поки людина не відповість.
2. Live: питання без жодних слідів у профілі → AI природно просить відповісти (не шаблон «Я не знаю відповіді з профілю…»).
3. Live: питання з чіткими даними в профілі → відповідь третьою особою без `needsHuman`.
4. Після підтвердження людиною — Arbiter може дати `NEXT_QUESTION` / `CLARIFY` за звичайними правилами.

## Критерії готовності

- [ ] Промпт Candidate Live вимагає третю особу і забороняє першу як відповідь за кандидата.
- [ ] Промпт описує три режими ANSWER з `needsHuman` для двох deferral-кейсів.
- [ ] Приклади в промпті не спонукають до дослівного копіювання старої фрази.
- [ ] Arbiter WAIT покриває assumption + confirm.
- [ ] Юніт-тести на наявність ключових формулювань у промптах зелені.
