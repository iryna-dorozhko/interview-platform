# Дизайн: Тестовий HR через seed

## Контекст

Day 1 bootstrap вимагає наявності тестового HR-користувача в базі після `db:seed`, щоб розблокувати подальшу розробку auth і HR-кабінету.  
У репозиторії вже є `backend/prisma/seed.js` і скрипт `npm run db:seed`; цей дизайн формалізує очікувану поведінку і межі змін.

## Мета

Забезпечити ідемпотентне створення одного стабільного тестового HR-акаунта при кожному запуску seed, без змін у схемі Prisma чи API.

## Архітектура

- Єдина точка сидінгу: `backend/prisma/seed.js`.
- Джерело істини для тестових даних — константи в цьому файлі (email, password, role).
- Створення користувача через `prisma.user.upsert` по унікальному полю `email`.
- Пароль хешується SHA-256 перед записом у `passwordHash`.
- Схема Prisma, міграції та auth API не змінюються.

## Тестові дані

| Поле | Значення |
|------|----------|
| email | `hr@test.com` |
| password (plain) | `123456` |
| role | `HR` |

> Лише для локальної розробки. Не використовувати в production.

## Потік даних

1. Розробник запускає `npm run db:seed` у `backend/` (або `npm --workspace backend run db:seed` з кореня).
2. Prisma викликає `node prisma/seed.js` (налаштовано в `backend/package.json`).
3. Seed підключається до PostgreSQL через `DATABASE_URL` або локальний fallback.
4. Пароль `123456` хешується SHA-256 → `passwordHash`.
5. `upsert` по `email: "hr@test.com"`:
   - якщо запису немає — `create` з `role: HR`;
   - якщо запис є — `update` `passwordHash` і `role` (повторний seed відновлює очікуваний стан).
6. У консоль виводиться: `Seeded HR user: hr@test.com`.

## Обробка помилок

- Помилка підключення до БД або Prisma → `console.error` + `process.exit(1)`.
- У `finally` — `prisma.$disconnect()`.
- Якщо міграції не застосовані — seed падає з явною помилкою Prisma (без silent fallback).

## Тестування і верифікація

**Ручна перевірка:**
- `cd backend && npm run db:seed` → очікується `Seeded HR user: hr@test.com`.
- Повторний запуск seed → той самий результат, без дублікатів (ідемпотентність).

**Опційно (smoke-тест):**
- Після seed у таблиці `User` є рівно один запис з `email = hr@test.com` і `role = HR`.

## Поза scope

- ENV-перевизначення email/password.
- Додаткові тестові HR-акаунти.
- Seed-дані для інтерв'ю, кандидатів, профілів.
- Зміни в схемі Prisma або auth API.

## Обґрунтування

Мінімальний підхід (один фіксований акаунт через `upsert`) дає передбачуваний bootstrap для локальної розробки без зайвої складності. Ідемпотентність дозволяє безпечно перезапускати seed під час розробки.
