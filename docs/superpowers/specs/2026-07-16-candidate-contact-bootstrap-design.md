# Candidate Contact Bootstrap Design

**Дата:** 2026-07-16  
**Статус:** Затверджено в brainstorming  
**Scope:** Початковий блок знайомства в `Candidate Agent` + збереження контактів у `CandidateProfile`

---

## Мета

Додати на старті діалогу кандидата короткий блок знайомства, щоб зібрати контактні дані для профілю кабінету:

- `fullName` (обов'язково)
- `email` (бажано підтвердити email реєстрації; fallback на акаунтний email)
- `phone` (опційно після 2-ї відмови)

Після цього агент переходить до чинного flow анкетування: досвід, сильні сторони, зони росту, цілі.

---

## Узгоджені правила діалогу

1. На першому кроці агент представляється.
2. Агент просить ім'я кандидата.
3. Агент просить email як додатковий спосіб зв'язку і пропонує залишити email, вказаний під час реєстрації.
4. Якщо кандидат не дає email:
   - агент робить одне коротке уточнення;
   - якщо кандидат далі відмовляється, у збереженні використовується `req.user.email` (fallback).
5. Агент просить телефон.
6. Якщо кандидат не дає телефон:
   - агент ставить повторний запит з поясненням, для чого номер;
   - якщо повторна відмова, анкету не блокуємо і йдемо далі без номера.

## Дані та персистентність

### Prisma

Розширити `CandidateProfile` новими полями:

- `fullName String`
- `email String`
- `phone String?`

`phone` nullable, бо дозволена повторна відмова.

### Extraction контракт

Розширити JSON extraction:

```json
{
  "fullName": "Ім'я Прізвище",
  "email": "candidate@example.com",
  "phone": "+380...",
  "experience": ["..."],
  "skills": { "strong": ["..."], "growth": ["..."] },
  "goals": ["..."],
  "summary": "..."
}
```

Правила:

- `fullName` обов'язкове, непорожній рядок.
- `email` обов'язкове на виході extraction; якщо в стенограмі немає явного email, бекенд підставляє `req.user.email`.
- `phone` може бути `null`.
- Решта полів лишаються за поточним контрактом.

### Backend flow

`POST /api/candidate-prep/:interviewId/finish`:

1. Отримати історію діалогу.
2. Запустити extraction з новими полями.
3. Після parse зробити fallback:
   - якщо `email` порожній/невалідний -> взяти `req.user.email`.
4. `upsert` у `CandidateProfile` з новими полями.

## Зміни в промптах

### `candidate-agent.uk.ts`

- Додати pre-phase збору контактів перед блоком про досвід/сильні/слабкі/цілі.
- Явно описати правила повторного запиту:
  - телефон: другий раз з поясненням, потім дозволений skip;
  - email: запропонувати email з реєстрації, при відмові дозволити fallback.

### `candidate-profile-extraction.uk.ts`

- Додати вимогу повернути `fullName/email/phone` у JSON.
- Для `phone` дозволити `null`.

---

## Readiness і UX

- `READY:true` дозволений, коли зібрані основні 4 теми, `fullName` і доступний email (наданий кандидатом або fallback з акаунта).
- Відсутній `phone` не блокує завершення після повторного запиту.
- У `CandidateProfileView` додати блок "Контактні дані":
  - Ім'я
  - Email
  - Телефон (`—`, якщо `null`)

## Тести

Мінімальний набір:

1. `candidate-agent.test.ts`:
   - промпт містить стартове представлення;
   - є правила для повторного запиту телефону з поясненням;
   - є правило про email реєстрації.
2. Тести parse extraction:
   - валідний JSON з `phone`;
   - валідний JSON з `phone: null`;
   - email відсутній -> fallback застосовується у route-level логіці.
3. `candidate-prep.test.ts`:
   - `finish` зберігає `fullName/email/phone`;
   - `finish` підставляє `req.user.email`, якщо email не отримано в extraction.
4. UI:
   - `CandidateProfileView` показує контактний блок і `—` для порожнього телефону.

## Не в scope

- Нормалізація міжнародних форматів телефону через строгий regex.
- Окрема сутність для контактних даних.
- Зміна auth-моделі `User`.
