# Дизайн: Стартова структура Frontend/Backend (Day 1)

## Контекст

Scope проєкту та продуктові рішення вже визначають чистий старт MVP як `npm workspaces` monorepo з поділом на `frontend` і `backend`.  
Цей дизайн описує лише bootstrap-крок Day 1, який розблоковує подальшу реалізацію.

## Мета

Створити мінімальний і надійний скелет репозиторію, який:

- вводить `frontend` і `backend` як незалежні workspace-пакети;
- надає єдину кореневу поверхню команд для install/dev/build/lint;
- залишається свідомо легким (без повного scaffold застосунків на цьому етапі).

## Архітектура

Репозиторій організовано як `npm workspaces` monorepo з двома пакетами:

- `frontend` — майбутній пакет веб-інтерфейсу;
- `backend` — майбутній пакет API та серверної логіки.

Кореневий пакет виконує роль оркестратора спільних скриптів і резолюції workspace-залежностей.

## Компоненти та відповідальність

### Корінь (`package.json`)

- Оголошує `workspaces: ["frontend", "backend"]`.
- Експортує агреговані скрипти:
  - `dev` — делегує виконання `dev`-скриптам у воркспейсах;
  - `build` — делегує виконання `build`-скриптам у воркспейсах;
  - `lint` — делегує виконання `lint`-скриптам у воркспейсах.

### Пакет `frontend`

- Власний `package.json` з мінімальними метаданими та базовими скриптами.
- Локальний `README.md` з призначенням пакета та наступними bootstrap-кроками.

### Пакет `backend`

- Власний `package.json` з мінімальними метаданими та базовими скриптами.
- Локальний `README.md` з призначенням пакета та наступними bootstrap-кроками.

## Потік команд/даних

1. Розробник запускає `npm install` у корені репозиторію.
2. npm резолвить кореневі та workspace-залежності як єдину установку monorepo.
3. Розробник запускає кореневі команди (`npm run dev`, `npm run build`, `npm run lint`).
4. Кореневі скрипти диспетчеризують виконання в обидва workspace-пакети, забезпечуючи єдиний операційний workflow.

## Обробка помилок і запобіжники

- Відсутні workspace-скрипти мають падати з явною помилкою в терміналі.
- Жодної тихої поведінки no-op для критичних скриптів (`dev`, `build`, `lint`).
- Назви скриптів повинні лишатися узгодженими між коренем і воркспейсами, щоб уникати дрейфу оркестрації.

## Тестування і верифікація (Day 1)

Мінімальна перевірка після створення структури:

- `npm install` у корені проходить успішно;
- `npm run dev` у корені виконує `dev`-скрипти воркспейсів;
- `npm run build` у корені виконує `build`-скрипти воркспейсів.

Цей scope навмисно обмежено bootstrap-верифікацією; налаштування framework/runtime відкладено на наступні кроки.

## Поза scope

- Повний Vite scaffold для `frontend`.
- Повний Express/TypeScript scaffold для `backend`.
- Схема БД, міграції, seed-дані, runtime-інтеграції.
