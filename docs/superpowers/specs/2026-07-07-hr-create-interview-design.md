# Створення співбесіди HR — Day 8 Design Spec

**Дата:** 2026-07-07
**Статус:** Затверджено в brainstorming
**Контекст:** День 8 плану розробки (README) — «HR створює співбесіду і отримує код для кандидата»
**Мова:** Українська (UI, повідомлення про помилки)

---

## Контекст і мета

Дні 3–7 вже дають повний prep-флоу (логін → анкета → профіль → підтвердження), але він працює на **єдиній seed-співбесіді** (`joinCode = "TEST01"`), яка створюється хардкодом у `backend/src/seed/hr-interview.js`. Реального `POST /interviews` немає — HR не може самостійно почати новий процес найму.

Мета Дня 8: HR тисне кнопку **«Створити співбесіду»** → бекенд створює новий запис `Interview` (`status: DRAFT`) з унікальним 6-символьним кодом → код одразу показується на екрані. Це стартова точка для подальшого prep-флоу (Дні 4–7, без змін) — той самий `interviewId` далі використовується для анкети, профілю та підтвердження.

**Важливий архітектурний нюанс.** `CompanyProfile.interviewId` — обов'язковий unique FK (`backend/prisma/schema.prisma:80`), тобто `Interview` мусить існувати ДО появи профілю компанії. Тому «Створити співбесіду» логічно є **першим кроком** нового процесу (замінює seed), а не фінальним кроком після підтвердження профілю. Перехід `DRAFT → AWAITING_CANDIDATE` при підтвердженні профілю вже реалізований у Дні 7 (`POST /prep/:interviewId/confirm`) і не потребує змін — Day 8 лише додає точку входу, яка породжує нові `Interview` замість покладання на seed.

Обмежень на кількість співбесід, які HR може створити, немає — множинність уже підтримується `GET /interviews/mine`, і буде повністю розкрита в дашборді Дня 9.

---

## Рішення з brainstorming

1. **Порядок дій:** «Створити співбесіду» створює порожню співбесіду (без профілю) одразу з кодом і статусом `DRAFT`. HR потім переходить в анкету компанії для цієї конкретної співбесіди — існуючий флоу Днів 4–7 не змінюється.
2. **Без ліміту на кількість:** HR може створювати необмежену кількість співбесід (в тому числі кілька в статусі `DRAFT` одночасно).
3. **Алфавіт коду:** `23456789ABCDEFGHJKLMNPQRSTUVWXYZ` (32 символи, без `0/O/1/I`) — код легше диктувати кандидату голосом чи телефоном без плутанини.
4. **Генерація:** `crypto.randomInt` для кожного з 6 символів (без нових npm-залежностей, консистентно з `backend/src/auth/password.ts`, де вже використовується `node:crypto`).
5. **Колізії:** retry-цикл (до 5 спроб) при `P2002` (унікальність `joinCode`) — імовірність колізії мізерна (32^6 ≈ 1.07 млрд варіантів), але обробляється явно.
6. **UI:** кнопка на `HomeView.vue` поруч з «Анкета компанії»; після успіху — банер на тій самій сторінці з кодом і кнопкою переходу в анкету (без окремої сторінки/модалки — консистентно з мінімалістичним стилем `HomeView`, буде замінено дашбордом Дня 9).

## API-контракт

```
POST /api/interviews
Headers: Authorization: Bearer <HR JWT>
Body:    (порожнє)
```

### Обробка запиту

1. `requireAuth` + `requireHr` (вже підключено на рівні router у `server.ts`, той самий router, що й `GET /interviews/mine`).
2. Згенерувати `joinCode` (6 символів з алфавіту вище).
3. Спробувати `prisma.interview.create({ data: { hrUserId: req.user.id, joinCode, status: "DRAFT" } })`.
   - При помилці `P2002` (унікальність `joinCode`) — згенерувати новий код і повторити, максимум 5 спроб.
   - Якщо всі спроби вичерпано — `500 { error: "Failed to generate unique join code" }`.
4. Відповідь `201`:

```json
{
  "interview": {
    "id": "cmr9...",
    "joinCode": "K7M2P9",
    "status": "DRAFT",
    "createdAt": "2026-07-07T10:00:00.000Z"
  }
}
```

### Помилки

| Ситуація | Код | Тіло |
|---|---|---|
| Немає токена / не HR | 401 / 403 | `{ error }` |
| Вичерпано спроби генерації унікального коду | 500 | `{ error: "Failed to generate unique join code" }` |

## Зміни в існуючих файлах

### `backend/src/utils/joinCode.ts` (новий файл)

```ts
import crypto from "node:crypto";

const ALPHABET = "23456789ABCDEFGHJKLMNPQRSTUVWXYZ";
const CODE_LENGTH = 6;

export function generateJoinCode(): string {
  let code = "";
  for (let i = 0; i < CODE_LENGTH; i++) {
    code += ALPHABET[crypto.randomInt(ALPHABET.length)];
  }
  return code;
}
```

### `backend/src/routes/interviews.ts`

Додається `router.post("/interviews", ...)` поруч з існуючим `GET /interviews/mine`, з retry-циклом навколо `generateJoinCode()` + `prisma.interview.create`. Перевірка Prisma-помилки унікальності — за кодом `error.code === "P2002"` (стандартний Prisma-патерн, аналогічно існуючим перевіркам помилок у `prep.ts`).

Роутинг у `server.ts` змін не потребує — `createInterviewsRouter` вже підключений з `requireAuth, requireHr`.

## Frontend (`HomeView.vue`, `api/interviews.ts`)

### API-клієнт (`frontend/src/api/interviews.ts`)

Нова функція поруч із `fetchMyInterviews`:

```ts
export type Interview = {
  id: string;
  joinCode: string;
  status: string;
  createdAt?: string;
};

export async function createInterview(): Promise<Interview> {
  const response = await fetchWithAuth("/api/interviews", { method: "POST" });
  if (!response.ok) {
    throw await parseError(response, "Не вдалося створити співбесіду");
  }
  const body = await response.json();
  return body.interview;
}
```

### UI (`HomeView.vue`)

- Нова кнопка **«Створити співбесіду»** (`btn-primary`) поруч із «Анкета компанії».
- Клік → виклик `createInterview()` → при успіху показується банер:

```
Співбесіду створено! Код для кандидата: K7M2P9
[Перейти до анкети →]
```

  де кнопка «Перейти до анкети» веде на `/prep/:interviewId` щойно створеної співбесіди (`router.push({ name: "company-prep", params: { interviewId } })`).
- Помилки виводяться через новий локальний `createError`-стан (за аналогією з наявним `prepNavError`).
- Банер лишається видимим до перезавантаження сторінки / створення нової співбесіди (тимчасове рішення до дашборду Дня 9, який покаже повний список з кодами).

## Тестування і верифікація

**Backend (`interviews.test.ts`), нові кейси:**
- `POST /interviews` повертає `201`, `joinCode` довжиною 6, лише символи з дозволеного алфавіту.
- Два послідовні виклики повертають різні `joinCode` (детермінований fake-prisma або мокнутий `crypto.randomInt`).
- Симуляція колізії: fake Prisma кидає `P2002` один раз → другий виклик `create` успішний → відповідь `201`.
- Вичерпання спроб (fake Prisma завжди кидає `P2002`) → `500`.
- Без токена / не-HR → `401` / `403`.
- Нова співбесіда з'являється в `GET /interviews/mine` для того самого HR.

**Ручний сценарій (Day 8 DoD):**
1. Логін HR → натиснути «Створити співбесіду».
2. Бачимо 6-символьний код на екрані.
3. Створити другу співбесіду → переконатись, що код інший.
4. Перевірка в БД: нова `Interview` зі статусом `DRAFT`, `hrUserId` відповідає залогіненому HR.
5. Перейти в анкету щойно створеної співбесіди → пройти чат → підтвердити профіль (Дні 4–7) → статус переходить у `AWAITING_CANDIDATE` (регресійна перевірка, що існуючий флоу працює на новостворених співбесідах, а не лише на seed).

**Build:** `npm run build` у корені без помилок.

## Поза scope (Day 8)

- Дашборд зі списком усіх співбесід і кодів (`GET /interviews/mine` вже повертає потрібні поля — рендер списку в UI належить Дню 9).
- Join кандидата за кодом — ендпоінт валідації `joinCode` з боку кандидата не реалізується.
- Видалення/архівація `DRAFT`-співбесід без профілю.
- Ліміти/квоти на кількість співбесід одного HR.
- Редагування чи повторна генерація `joinCode` для вже створеної співбесіди.
