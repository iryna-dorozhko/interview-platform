# Candidate Live: відповідь на всі відкриті питання після перебивання HR

**Дата:** 2026-07-23  
**Статус:** затверджено в чаті (підхід B: heuristic + промпт)

## Проблема

На live-співбесіді типовий сценарій:

1. `AGENT_COMPANY` ставить змістовне питання (напр. про REST API).
2. До відповіді кандидата `HUMAN_HR` перебиває іншим питанням (напр. про проєкти).
3. Arbiter дає `ANSWER`.
4. Candidate Agent відповідає лише на останнє (HR) питання; часто ще й коротко «Кандидат уже розповів…», якщо тема вже була в чаті.
5. Питання Company лишається без відповіді.

Корінь: ANSWER-nudge зараз перелічує **усі** історичні повідомлення Company/HR («не дублюй»), а промпт орієнтує на «поточне» питання — модель природно бере лише останнє. Немає явного списку **відкритих** питань і правила відповісти на всі в одному повідомленні.

## Цілі

- При `ANSWER`, якщо після останньої відповіді кандидата є кілька повідомлень від Company/HR без відповіді — Candidate публікує **одне** повідомлення, що покриває **усі** відкриті пункти (на практиці зазвичай ≤2).
- Порядок відповідей — хронологічний (як з’явились у чаті).
- Якщо тема вже повністю прозвучала в чаті — коротке підтвердження без деталей; для ще незакритих — повна відповідь з профілю.
- `confidence` лишається одним полем; окремої агрегації (worst-case тощо) немає — модель обирає як зараз.

## Нецілі

## Підхід

**B — heuristic у коді + оновлення промпту/nudge.**

1. Детерміновано зібрати відкриті питання інтерв’юерів.
2. Передати їх у ANSWER-nudge з явною інструкцією відповісти на всі в одному повідомленні.
3. У system prompt зафіксувати правила для кількох відкритих і короткого ack на вже покрите.

## Евристика «відкрите питання»

Нова функція (заміна або уточнення `collectRecentInterviewerQuestions` для ANSWER):

`collectOpenInterviewerQuestions(history)`:

- Знайти індекс останнього повідомлення з `authorType` ∈ {`AGENT_CANDIDATE`, `HUMAN_CANDIDATE`}.
- Узяти всі наступні повідомлення з `authorType` ∈ {`AGENT_COMPANY`, `HUMAN_HR`} з непорожнім `content`.
- Порядок зберегти хронологічний.

Якщо кандидата ще не було в чаті — усі Company/HR з історії вважаються відкритими (як зараз для першого ходу).

Ця евристика навмисно проста: «відкрите» ≈ інтерв’юер після останньої відповіді кандидата. Достатньо для сценарію перебивання; не намагаємось NLP-класифікувати «чи це питання».

## Поведінка Candidate Live Agent

При `ANSWER` і непорожньому списку відкритих:

| Ситуація | Поведінка в одному `message` |
|---|---|
| Одне відкрите | Як зараз: відповідь з профілю / inferred / unknown |
| Кілька відкритих | Відповіді на **всі**, хронологічно, в одному повідомленні |
| Відкрите, але відповідь уже повністю була в чаті | Коротке підтвердження без повторення деталей |
| Відкрите і ще не покрите | Повна відповідь з профілю (режими confirmed / inferred / unknown) |
| Змішані | Короткий ack на вже покриті + повні відповіді на нові |

`confidence`: одне значення на повідомлення; модель обирає сама (без примусу worst-case).

## Nudge / промпт

**ANSWER nudge block** (замість «усі питання Company/HR у чаті»):

- Заголовок на кшталт: «Відкриті питання (відповідай на всі в одному повідомленні, хронологічно):»
- Нумерований список з `collectOpenInterviewerQuestions`.

**System prompt (`candidate-live-agent.uk.ts`):**

- Додати правило: кілька відкритих → одне повідомлення на всі.
- Уточнити правило «вже в чаті»: коротке підтвердження **лише для того пункту**, не замість відповіді на інші відкриті.

`CANDIDATE_QUESTIONS` nudge може лишитись зі списком «не дублюй уже поставлені» (історичний список для anti-repeat) — поза критичним path цього spec; мінімальна зміна: не ламати існуючі тести anti-repeat, або розвести open vs historical helpers явно.

## Тест-план (авто)

1. Після `AGENT_CANDIDATE` з’являються Company + HR → open = обидва тексти.
2. Питання **до** відповіді кандидата не входять у open.
3. ANSWER-nudge містить обидва відкриті + формулювання про відповідь на всі в одному повідомленні.
4. Промпт містить правило multi-open і коротке підтвердження для вже сказаного.

## Тест-план (ручний)

1. Live: Company питає про REST → HR перебиває про проєкти (якщо проєкти вже були вище) → Candidate одним повідомленням: коротке ack про проєкти + змістовна відповідь про REST.
2. Одне відкрите питання без перебивання — поведінка без регресії.
