# Пріоритетність вимог вакансії у match score

## Мета

Дати HR змогу під час анкетування поділити вимоги вакансії на критичні та бажані, а потім контрольовано врахувати цей поділ у відсотку відповідності кандидата. Результат має бути пояснюваним для HR, стабільним на рівні бізнес-правил і сумісним із наявними вакансіями.

## Узгоджені продуктові правила

- Є два рівні: **критичні** та **бажані** вимоги.
- Company Agent окремо збирає обидва списки, а HR перевіряє та редагує їх перед підтвердженням профілю вакансії.
- Явно невиконана критична вимога обмежує фінальний score до 69%.
- Відсутність даних про критичну вимогу має статус `unknown`: вона знижує score, але не активує cap.
- Детальний breakdown бачить лише HR. Кандидат бачить тільки відсоток.
- Усі вимоги старих вакансій трактуються як бажані.

## Модель даних

`CompanyProfile.requirements` залишається колонкою `Json`, але новий канонічний формат має вигляд:

```ts
type VacancyRequirements = {
  critical: string[];
  desired: string[];
};
```

Сервісний parser нормалізує старий `string[]` у:

```ts
{ critical: [], desired: legacyRequirements }
```

Обидва списки не можуть бути порожніми одночасно. Кожен із них окремо може бути порожнім. Порожні рядки видаляються, порівняння дублікатів не враховує регістр. Якщо одна вимога є в обох списках, вона залишається лише у `critical`.

`VacancyMatchScore` розширюється полями:

- `breakdown Json` — оцінки вимог, докази, context score і інформація про cap;
- `rankedForVacancyConfirmedAt DateTime` — версія підтвердженого профілю вакансії.

Унікальність актуального score визначається кандидатом, вакансією, `CandidateProfile.confirmedAt` і `CompanyProfile.confirmedAt`.

`VacancyApplication` отримує `matchBreakdown Json`, щоб зберігати snapshot пояснення на момент подання заявки. Подальший перерахунок match score не змінює історичну заявку.

## Анкетування вакансії

Company Agent проходить вимоги двома окремими кроками:

1. Просить назвати критичні вимоги, без яких кандидат не вважається повністю відповідним.
2. Просить назвати бажані вимоги, які підсилюють відповідність, але не є блокувальними.

Агент не повинен самостійно підвищувати чи знижувати пріоритет, якщо HR назвав категорію явно. Extraction prompt повертає:

```json
{
  "requirements": {
    "critical": ["..."],
    "desired": ["..."]
  }
}
```

Після завершення чату HR бачить дві окремі редаговані секції — «Критичні вимоги» і «Бажані вимоги». Дані, підтверджені HR у цій формі, є остаточним джерелом істини.

## Семантична оцінка LLM

LLM не визначає фінальний відсоток. Для кожної переданої вимоги вона повертає:

```ts
type RequirementAssessment = {
  requirement: string;
  priority: "critical" | "desired";
  status: "met" | "unknown" | "unmet";
  evidence: string;
};
```

Семантика статусів:

- `met` — профіль кандидата містить достатній доказ відповідності;
- `unknown` — у профілі недостатньо даних для висновку;
- `unmet` — профіль містить явний доказ невідповідності.

LLM також повертає `contextFit` у діапазоні 0–100. Він оцінює релевантність ролі, досвіду, цілей, очікувань і культури, не повторюючи оцінку окремих вимог.

`evidence` — коротке фактичне пояснення українською на основі профілю кандидата. Воно не повинно містити прихованих міркувань моделі або вигаданих фактів.

Parser вимагає, щоб кожна вхідна вимога була оцінена рівно один раз. Невідомі або додані моделлю вимоги, дублікати, заборонені статуси, порожній evidence та невалідний `contextFit` роблять всю відповідь невалідною.

## Детермінована формула

Статуси переводяться в числові значення:

- `met = 100`;
- `unknown = 50`;
- `unmet = 0`.

Далі бекенд обчислює:

1. `criticalFit` — середнє значення оцінок критичних вимог.
2. `desiredFit` — середнє значення оцінок бажаних вимог.
3. Якщо обидві категорії непорожні:
   `requirementsFit = 0.75 × criticalFit + 0.25 × desiredFit`.
4. Якщо одна категорія порожня, `requirementsFit` дорівнює fit наявної категорії без штучного штрафу.
5. Якщо вимог немає, фінальний результат базується лише на `contextFit`.
6. За наявності вимог:
   `rawScore = 0.80 × requirementsFit + 0.20 × contextFit`.
7. Якщо хоча б одна критична вимога має статус `unmet`:
   `finalScore = min(round(rawScore), 69)`.
8. Інакше:
   `finalScore = round(rawScore)`.
9. Результат додатково обмежується діапазоном 0–100.

Розрахунок реалізується чистою функцією без доступу до LLM або бази даних.

## Match flow і кешування

1. Сервіс завантажує підтверджений профіль кандидата та всі підтверджені вакансії.
2. Для кожної вакансії шукає score з поточними версіями профілю кандидата й вакансії.
3. Актуальні cached scores використовуються одразу.
4. Лише вакансії без актуального score передаються LLM одним пакетом.
5. Після повної валідації відповіді бекенд рахує score і атомарно зберігає нові записи.
6. Під час подання заявки `matchScore` і `breakdown` копіюються у `VacancyApplication`.

Це усуває поточну проблему, коли наявність хоча б одного cached score припиняє оцінювання всіх інших вакансій. Нова підтверджена вакансія зможе з’явитися у match results без повторного підтвердження анкети кандидата.

Якщо LLM повертає невалідну або неповну відповідь, новий пакет scores не зберігається, а API повертає наявну помилку `MATCH_UNAVAILABLE`. Уже валідні cached scores не видаляються й не змінюються.

## API та UI

Vacancy prep API приймає і повертає структурований `requirements`. PATCH профілю застосовує ті самі правила нормалізації та валідації, що й extraction parser.

У `VacancyPrepView` замість одного textarea «Вимоги» з’являються:

- «Критичні вимоги»;
- «Бажані вимоги».

Candidate match API не повертає breakdown. HR application details API повертає snapshot breakdown лише для HR-власника вакансії.

У деталях заявки HR бачить:

- критичні та бажані вимоги в окремих секціях;
- статус «Відповідає», «Не підтверджено» або «Не відповідає»;
- короткий evidence;
- окреме повідомлення, якщо score обмежено до 69% через критичну вимогу.

## Безпека та доступ

- Breakdown не включається у candidate endpoints.
- HR отримує breakdown лише для вакансій, що належать його користувачу.
- LLM оцінює лише надані профілі та не має права додавати факти чи вимоги.
- Evidence є стислим посиланням на профіль, а не chain-of-thought.

## Тестування

### Unit

- Нормалізація нового й legacy форматів вимог.
- Видалення порожніх значень і дублікатів; перевага `critical`.
- Parser assessment: повний набір вимог, статуси, evidence, `contextFit`.
- Формула: ваги, одна порожня категорія, відсутність вимог, `unknown`, cap 69, округлення та межі.

### Service та routes

- Часткове використання кешу й оцінка лише відсутніх вакансій.
- Поява нової підтвердженої вакансії без перепідтвердження кандидата.
- Інвалідація після нового `confirmedAt` кандидата або вакансії.
- Відсутність частково записаного пакета після помилки LLM.
- Редагування й підтвердження двох категорій вимог.
- Snapshot breakdown у заявці.
- HR authorization і відсутність breakdown у candidate response.

### Frontend

- Редагування двох списків у профілі вакансії.
- Відображення трьох статусів і cap-повідомлення у деталях заявки HR.
- Перевірка, що candidate UI продовжує отримувати й показувати тільки score.

Перед завершенням реалізації мають пройти backend tests, frontend build, lint і typecheck, доступні в репозиторії.

## Поза межами

- Числові ваги, які HR задає вручну для кожної вимоги.
- Більше двох рівнів пріоритетності.
- Жорстке виключення вакансії з видачі.
- Відображення breakdown кандидату.
- Повністю детермінований словниковий matcher без LLM.
