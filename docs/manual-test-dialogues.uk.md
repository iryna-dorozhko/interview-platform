# Готові відповіді для ручної перевірки

Скрипт для копіювання під час тестування. Усі тексти узгоджені між собою: профіль HR-анкети, профіль кандидата й live-співбесіда описують одну й ту саму вакансію **Middle Backend Developer**.

**Тестові акаунти:** `hr@test.com` / `123456`, кандидат — `candidate@test.com` / `123456` (реєстрація), код співбесіди з seed — `TEST01`.

> Агент ставить по одному питанню за раз. Надсилай відповіді **по черзі** (1 → 2 → 3 …). Якщо агент уточнює тему — можна повторити відповідь з того ж блоку або надіслати «Додатково» з того ж розділу.

---

## 1. HR — анкета компанії (окремий чат)

**Де:** `/vacancies/:id/prep`  
**Хто пише:** HR (`hr@test.com`)  
**Після чату:** «Завершити чат» → «Підтвердити профіль»

### Основний сценарій (мінімум 4 відповіді)

**Відповідь 1 — посада**

```
Шукаємо Middle Backend Developer у продуктову команду платіжного сервісу. Команда з п'яти розробників: два backend, два frontend, один QA. Людина працюватиме над API для B2B-клієнтів.
```

**Відповідь 2 — вимоги**

```
Обов'язково: TypeScript, Node.js, PostgreSQL, досвід з REST API від трьох років. Бажано: Docker, базове розуміння AWS, досвід з мікросервісною архітектурою. Англійська — на рівні читання документації.
```

**Відповідь 3 — культура**

```
У нас плоска структура без зайвої бюрократії. Працюємо remote-first: більшість команди в Києві та Львові, синхронні зустрічі двічі на тиждень. Цінуємо відкриту комунікацію в Slack, code review і відповідальність за свій сервіс.
```

**Відповідь 4 — очікування**

```
Перший місяць — онбординг, ознайомлення з кодовою базою, перші невеликі таски з ментором. До третього місяця очікуємо, що людина самостійно веде один мікросервіс і бере участь у плануванні спринту.
```

**Відповідь 5 — якщо агент просить уточнення (опційно)**

```
Так, стек фіксований: Node.js 20, PostgreSQL 15, Redis для кешу. Команда використовує GitHub Actions для CI/CD.
```

### Очікуваний результат

- Після 3+ обмінів з'являється кнопка «Завершити чат».
- Після finish — профіль з полями `role`, `requirements`, `culture`, `expectations`.
- Після confirm — анкета в статусі «Підтверджена».

## 2. Кандидат — анкета (окремий чат)

**Де:** `/candidate/prep/:interviewId`  
**Хто пише:** кандидат (`candidate@test.com`)  
**Після чату:** «Завершити чат» → «Підтвердити профіль»

### Основний сценарій (мінімум 4 відповіді)

**Відповідь 1 — досвід**

```
Три роки працюю backend-розробником у FinTech-стартапі. Основні проєкти: платіжний шлюз для e-commerce і внутрішня система звірки транзакцій. Стек: TypeScript, Node.js, PostgreSQL, Redis.
```

**Відповідь 2 — сильні сторони**

```
Добре знаю TypeScript і проєктую REST API. Оптимізував повільні запити до PostgreSQL — середній час відповіді API знизився з 800 мс до 120 мс. Комфортно працюю з code review і документацією.
```

**Відповідь 3 — зони росту**

```
Публічні виступи даються важко — на мітапах виступав двічі і хвилювався. DevOps знаю на базовому рівні: Docker-compose для локальної розробки, але Kubernetes ще вивчаю.
```

**Відповідь 4 — цілі**

```
Хочу перейти на рівень senior у продуктовій команді, де видно вплив на бізнес. Шукаю стабільну команду з менторством і можливістю розвивати архітектурні навички.
```

**Відповідь 5 — якщо агент просить уточнення (опційно)**

```
Останній проєкт — інтеграція з трьома платіжними провайдерами. Відповідав за API-шар і міграції БД. Команда була з чотирьох backend-розробників.
```

### Очікуваний результат

- Після 3+ обмінів — «Завершити чат».
- Профіль містить `experience`, `skills.strong`, `skills.growth`, `goals`, `summary`.
- Після confirm — кандидат готовий до join за кодом.

### Перевірка контактних даних (профіль кандидата)

**Де:** `/candidate/profile` після «Завершити чат»

- Блок «Контактні дані» містить ім'я, email і телефон з профілю.
- Якщо `phone` відсутній (`null`) — у полі «Телефон» відображається «—».

## 3. Спільний live-чат (співбесіда)

**Де:** HR — `/interviews/:id/room`, кандидат — `/candidate/interview` → «Увійти в співбесіду»  
**Умова:** обидва профілі підтверджені, статус `READY` → обидва в кімнаті → `LIVE`  
**Хто пише:** HR і кандидат по черзі у своїх вкладках

Після кожного повідомлення людини чекай ~2.5 с — спрацює ланцюжок **Arbiter → Company → Candidate (AI)**.

### 3.1 Вітання та старт

| Крок | Хто | Текст для копіювання |
|------|-----|----------------------|
| 1 | HR | `Доброго дня! Мене звати Олена, я HR у команді. Рада вас бачити на співбесіді.` |
| 2 | Кандидат | `Доброго дня! Мене звати Андрій. Теж радий приєднатися, дякую за запрошення.` |
| 3 | — | Зачекай: Arbiter може написати на кшталт «Давайте почнемо співбесіду», потім Company поставить питання, Candidate (AI) відповість з профілю. **Нічого не надсилай**, поки агенти не відпрацюють. |

### 3.2 HR веде діалог (основний сценарій)

| Крок | Хто | Текст для копіювання |
|------|-----|----------------------|
| 4 | HR | `Розкажіть, будь ласка, про найскладнішу технічну задачу, яку ви вирішували на останньому проєкті?` |
| 5 | — | Candidate (AI) відповість з профілю. Якщо відповіді в профілі немає — AI попросить живого кандидата відповісти; тоді надішли крок 6. |
| 6 | Кандидат *(лише якщо AI попросив)* | `Найскладніше було звести разом три платіжні API з різними форматами помилок. Я зробив єдиний адаптер і додав retry-логіку — після релізу відмови платежів зменшились на 15%.` |
| 7 | HR | `Як ви зазвичай підходите до code review у команді?` |
| 8 | Кандидат *(якщо AI не відповів або питання поза профілем)* | `Намагаюсь давати конструктивні коментарі: пояснюю чому, пропоную альтернативу. Сам при рев'ю очікую такого ж підходу — це допомагає швидше вчитися.` |
| 9 | HR | `У нас remote-first і багато асинхронної комунікації. Як вам такий формат?` |
| 10 | Кандидат | `Мені підходить. Останні два роки працював віддалено, звик до Slack і коротких статус-апдейтів. Важливо мати чіткі дедлайни в тікетах — тоді async працює добре.` |
| 11 | HR | `Є питання з нашого боку — дякую за відповіді. Ми зв'яжемося з вами щодо наступних кроків.` |

### 3.3 Короткий сценарій (якщо мало часу)

| Крок | Хто | Текст |
|------|-----|-------|
| 1 | HR | `Вітаю! Почнемо, коли будете готові.` |
| 2 | Кандидат | `Доброго дня! Готовий.` |
| 3 | — | Дочекайся Arbiter + Company + Candidate (AI) |
| 4 | HR | `Чому вас зацікавила саме backend-роль у продуктовій команді?` |
| 5 | Кандидат | `Хочу бачити вплив коду на продукт і працювати ближче до бізнес-вимог, ніж на аутсорсі.` |

### 3.4 Перевірка Arbiter (опційно)

**Офтоп** — надішли від HR або кандидата, щоб Arbiter модерував:

```
Хтось бачив вчорашній футбол? Дуже драматичний матч.
```

**Очікування:** Arbiter може м'яко повернути розмову до теми співбесіди.

**Завершення** — коли основні питання вичерпано, HR:

```
Думаю, на сьогодні достатньо. Дякую за співбесіду!
```

**Очікування:** Arbiter може запропонувати завершити співбесіду.

## 4. Швидка шпаргалка

| Етап | URL | Скільки повідомлень | Дія після чату |
|------|-----|---------------------|----------------|
| HR-анкета | `/vacancies/:id/prep` | 4–5 | Finish → Confirm |
| Анкета кандидата | `/candidate/prep/:id` | 4–5 | Finish → Confirm |
| Live-чат | `/interviews/:id/room` + `/candidate/interview` | 2 вітання + 2–4 питання | HR: «Завершити співбесіду» |

**Порядок повного E2E:**

1. HR: анкета → confirm  
2. HR: створити співбесіду → скопіювати код  
3. Кандидат: join за кодом → анкета → confirm  
4. Обидва: увійти в кімнату → тексти з розділу 3  
5. HR: завершити співбесіду (кнопка при `LIVE`)
